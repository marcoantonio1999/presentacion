### REM-HIGH-02: Implementar Certificate Pinning

**Origen:** [HIGH-05]
**Owner:** AMXNetworkSdk Team + Security Lead
**Esfuerzo:** 2-3 días (implementación + pin rotation strategy)
**Riesgo de Regresión:** ALTO — requiere gestión de pin rotation

```kotlin
// En el módulo DI de AMXNetworkSdk — NetworkModule.kt:
@Provides
@Singleton
fun provideOkHttpClient(
    interceptors: List<Interceptor>
): OkHttpClient {
    val certificatePinner = CertificatePinner.Builder()
        // Obtener pins ejecutando: openssl s_client -connect mfwkstbzte-api.clarovideo.net:443 | 
        //   openssl x509 -pubkey -noout | openssl pkey -pubin -outform der | 
        //   openssl dgst -sha256 -binary | openssl enc -base64
        .add("mfwkstbzte-api.clarovideo.net", "sha256/AAAA...=")  // Pin primario
        .add("mfwkstbzte-api.clarovideo.net", "sha256/BBBB...=")  // Pin backup (CA intermedia)
        .build()

    return OkHttpClient.Builder()
        .certificatePinner(certificatePinner)
        .addInterceptor(/* auth interceptor */)
        .build()
}
```

> 3. Documentar el proceso de rotación en el runbook de ops

## 🔴 HALLAZGOS CRÍTICOS

### [CRIT-01] Tráfico HTTP Cleartext Habilitado en Producción
**Severidad:** CRÍTICO | **OWASP Mobile:** M5 - Insecure Communication | **CWE:** CWE-319

**Evidencia directa** (`app/src/main/res/xml/network_security_config.xml`):
```xml
<network-security-config>
    <base-config cleartextTrafficPermitted="true">   <!-- ← CRÍTICO -->
        <trust-anchors>
            <certificates src="system" />
            <certificates src="@raw/charles_ssl_cert" />  <!-- ← CRÍTICO -->
            <certificates src="user" />                    <!-- ← ALTO -->
        </trust-anchors>
    </base-config>
```

**Impacto:**
- `cleartextTrafficPermitted="true"` aplica a **todos los build variants** incluido release
- Toda comunicación HTTP (sin cifrar) es permitida — expone tokens, credenciales y PII en tránsito
- En entorno STB (red de TV de pago), el riesgo de interception por operadoras o ISPs es real
- El estándar de la industria para apps de streaming desde Android 9+ es cleartext=false por defecto

**Clasificación OWASP Top 10:** A02:2021 – Cryptographic Failures

---

### [CRIT-02] Certificado Charles Proxy en Build de Producción
**Severidad:** CRÍTICO | **OWASP Mobile:** M5 + M8 | **CWE:** CWE-295

**Evidencia directa** (`app/src/main/res/xml/network_security_config.xml`):
```xml
<base-config cleartextTrafficPermitted="true">
    <trust-anchors>
        <certificates src="@raw/charles_ssl_cert" />  <!-- Certificado de proxy en PRODUCCIÓN -->
    </trust-anchors>
</base-config>
<debug-overrides>
    <trust-anchors>
        <certificates src="@raw/charles_ssl_cert" />  <!-- Duplicado en debug-overrides también -->
    </trust-anchors>
</debug-overrides>
```

**Impacto:**
- El certificado de Charles Web Proxy (`@raw/charles_ssl_cert`) está confiado en `<base-config>`, no solo en `<debug-overrides>`
- En producción, **cualquier atacante con el certificado de Charles** puede realizar Man-in-the-Middle contra todos los usuarios de la aplicación
- Interceptación de: tokens de autenticación, credenciales de streaming, llamadas a API REST, datos de EPG
- Este es el hallazgo de mayor impacto del repositorio — afecta directamente la confidencialidad de millones de usuarios de Claro Video

**Clasificación OWASP Top 10:** A07:2021 – Identification and Authentication Failures

**Impacto:**
