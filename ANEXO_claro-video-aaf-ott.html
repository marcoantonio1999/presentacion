## 🎯 Resumen Arquitectónico

### Stack Tecnologico

React 15.6.1 + Redux 3.7.2 + Webpack 3.5.1. Bitmovin Player 8.165 + Shaka Player 3.0 para DRM. Express 4.16 + PM2. Node.js 10.24.1. TypeScript 3.9.5 (~8%).

### Patrones Arquitectonicos Principales

Component-Based Architecture + Flux/Redux unidireccional. Façade (AAF.js), Strategy (player), Container/Presentational, Lazy Loading (react-loadable).

### Principios de Diseno

1. Unidireccionalidad del flujo de datos (Redux Flux)
2. Separación entre UI y lógica de negocio
3. Módulos con responsabilidad única
4. Cleanup obligatorio en `componentWillUnmount` (Smart TV)
5. No asumir que `window` existe (WebWorker context)

## 🔧 Patrones de Diseño

### Patrón Arquitectónico

Component-Based + Flux/Redux unidireccional. El patrón dominante es Container/Presentational con `connect()` HOC para vincular UI con estado.

### Dependency Injection

Este proyecto **NO usa DI formal**. Usa: props drilling, Redux `connect()` como inyección de estado global, e importaciones directas de módulos.

### Repository Pattern

`src/requests/RequestManager.js` actúa como repositorio HTTP centralizado. Todas las acciones Redux que necesitan datos externos lo usan como única fuente.

### Observer/State Management

Redux como patrón Observer: los componentes se suscriben al store vía `connect()`. Los cambios de estado notifican automáticamente a todos los componentes suscritos.

| Patrón | Implementación | Fuente |
|--------|---------------|--------|
| **Flux / Redux** | Estado global unidireccional | `src/store.js`, `src/actions/`, `src/reducers/` |
| **Container/Presentational** | HOC `connect()` para inyección de estado | `src/containers/` |
| **Façade** | `AAF.js` expone interfaz unificada | `src/AAF/AAF.js` |
| **Strategy** | AAFPlayer selecciona Bitmovin vs Shaka | `src/AAFPlayer/AAFPlayer.js` |
| **Factory** | `factoryMaker.js` genera bundles por plataforma | `factoryMaker.js` |
| **Singleton** | `ChannelsSingleton` para datos de canales | `src/components/Epg/ChannelsSingleton.js` |
| **Event-Driven** | `AAFPlayerEvent` para eventos del player | `src/AAFPlayer/common/` |
| **Lazy Loading** | `react-loadable` para code splitting | `package.json:93` |

## 🔒 Seguridad y Consideraciones

### Autenticacion y Autorizacion

`src/authentication/` para flujos de auth de usuario. `AAF.Auth` para renovación de tokens. `AAF.Pin` para control parental. `src/2FA/` para segundo factor de autenticación.

### Manejo de Secrets

Todos los secrets en `.env` (nunca hardcodeados). Carga en `config/env.js`. Keys de Bitmovin y URLs de API configuradas en runtime vía variables de entorno.

### Validacion de Datos

> ⚠️ No se detectó capa centralizada de validación. La validación se realiza de forma distribuida en componentes individuales y reducers.

### Proteccion de APIs

Autenticación vía tokens gestionados por `DeviceStorage`. Las APIs DRM (Bitmovin/Shaka) usan Widevine y PlayReady para protección de contenido.

1. **Variables de entorno:** Usar `.env` para todos los secrets — nunca hardcodear en código
2. **DRM:** Bitmovin y Shaka gestionan protección DRM (Widevine, PlayReady) en runtime
3. **2FA:** Módulo en `src/2FA/` con `TimerManager.ts` y `callbacks.ts`
4. **Control parental:** `AAF.Pin` gestiona verificación de PIN
5. **Tokens:** Gestionados via `DeviceStorage` (abstracción de localStorage)
6. **Smart TV:** `window` puede no existir en Web Worker context — verificar antes de usar
7. **Memory leaks:** Siempre hacer cleanup en `componentWillUnmount` (crítico en Smart TV)

## 🎯 Principios Generales

### Restricciones del Stack (React 15.x)

> **⚠️ CRÍTICO:** Este proyecto usa **React 15.6.1**. Las siguientes restricciones son obligatorias:

| ❌ Prohibido | ✅ Usar en su lugar |

3. **Minimizar el bundle size** — los TV browsers tienen límites de memoria
4. **Evitar memory leaks** — event listeners y subscriptions deben limpiarse siempre

## 🚫 Anti-patrones a Evitar
### Que Evitar

Los siguientes patrones están prohibidos en este proyecto.

### Code Smells Comunes

Props drilling excesivo, lógica de negocio en presentacionales, fetch directo en componentes.

### Ejemplos de Mal Codigo

Ver subsecciones detalladas a continuación.

### Como Refactorizar

Mover lógica a containers y Redux actions; usar RequestManager para todas las llamadas HTTP.
### 1. Lógica de negocio en componentes visuales

```javascript
// ❌ MAL: lógica en componente presentacional
const VideoCard = ({ contentId }) => {
  const [data, setData] = useState(null); // ❌ No hay hooks en React 15
  fetch(`/api/content/${contentId}`).then(d => setData(d)); // ❌ Fetch directo
  return <div>{data && data.title}</div>;
};

// ✅ BIEN: lógica en container + Redux
class VideoCardContainer extends Component {
  componentDidMount() {
    this.props.fetchContent(this.props.contentId);
  }
  render() {
    return <VideoCard title={this.props.title} />;
  }
}
```

### 2. Mutar el estado de Redux directamente

```javascript
// ❌ MAL: mutación directa
case PLAY_MEDIA:
  state.isPlaying = true; // ❌ Mutación directa
  return state;

// ✅ BIEN: estado inmutable
case PLAY_MEDIA:
  return { ...state, isPlaying: true };
```

### 3. No hacer cleanup en Smart TV

```javascript
// ❌ MAL: memory leak en Smart TV
componentDidMount() {
  window.addEventListener('keydown', this.handleKey);
}
// Sin componentWillUnmount → memory leak

// ✅ BIEN: siempre limpiar
componentWillUnmount() {
  window.removeEventListener('keydown', this.handleKey);
  if (this.player) {
    this.player.destroy();
    this.player = null;
  }
}
```

### Adding New Features

Seguir el patrón Redux: 1) constante en `src/constants/`, 2) action creator en `src/actions/`, 3) reducer en `src/reducers/`, 4) componente en `src/components/`, 5) container en `src/containers/`, 6) tests.

### Modifying Existing Code

Verificar impacto en reducers conectados antes de modificar actions. Revisar `mapStateToProps` en containers afectados. Ejecutar `yarn test` tras cambios.

### Refactoring Guidelines

Priorizar: node-sass → sass (bajo riesgo), feature-based reducers (alto valor), TypeScript en requests (mejor tipado). NO romper contratos de API entre módulos.

Migración gradual por módulo/feature. Mantener compatibilidad con React 15 hasta que toda la base esté lista. Usar ramas de feature largas con CI continuo.

`claro-video-aaf-ott` (`stvreact v22.2.1`) es la SPA OTT de Claro Video para Smart TV y web, construida con React 15.6.1 + Redux 3.7.2 + Webpack 3.5.1. La aplicación incluye reproducción de video con DRM (Bitmovin + Shaka Player), guía EPG, autenticación y pagos.

## 🔴 Errores de instalación

### Environment Issues

Verificar Node.js v10.24.1 con `nvm use 10.24.1` y Yarn 1.22.22 antes de instalar.

### Dependency Problems

Problemas con dependencias: limpiar `node_modules` y ejecutar `yarn install --network-timeout 600000`.

### Configuration Errors

Verificar que existen los archivos `.env` necesarios según `config/env.js`.

### Permission Issues

En Windows, ejecutar terminal como Administrador si hay errores de permisos en `node_modules`.

### `ENOMEM` al ejecutar `yarn install`

**Síntoma:**
```
FATAL ERROR: CALL_AND_RETRY_LAST Allocation failed - JavaScript heap out of memory
```

**Causa:** Node.js no tiene suficiente memoria heap para instalar todas las dependencias.

**Solución:**
```bash
export NODE_OPTIONS="--max-old-space-size=4096"
yarn install --network-timeout 600000
```

---

### `node-sass` falla al compilar

**Síntoma:**
```
Error: Node Sass does not yet support your current environment:
Windows 64-bit with Unsupported runtime (XX)
```

**Causa:** `node-sass 4.5.3` requiere versión específica de Node.js.

**Solución:**
```bash
# 1. Asegurarte de usar Node v10.24.1
nvm use 10.24.1
node -v   # debe mostrar v10.24.1

# 2. Recompilar node-sass para tu plataforma
yarn run rebuild-sass
# o
node_modules/.bin/node-sass --version   # verificar compatibilidad
```

---

### `yarn install` tarda demasiado o se cuelga

**Síntoma:** Instalación se detiene sin error aparente.

**Solución:**
```bash
# Aumentar el timeout de red
yarn install --network-timeout 600000

# Si el problema persiste, limpiar caché de yarn
yarn cache clean
yarn install --network-timeout 600000
```

---

### Módulo no encontrado al instalar

**Síntoma:**
```
Cannot find module 'some-package'
```

**Solución:**
```bash
# Eliminar node_modules y reinstalar
rmdir /s /q node_modules    # Windows
# rm -rf node_modules       # macOS/Linux
yarn install --network-timeout 600000
```

## 🔵 Errores de build

### Build Failures

Errores en `yarn run compile`: verificar variables de entorno y memoria disponible.

### Dependency Conflicts

Conflictos de dependencias: no actualizar React, Webpack o Jest sin plan de migración.

### Version Mismatches

Verificar que Node.js es v10.24.1 (`node -v`); incompatibilidades causan fallos en sass y otros.

### Platform-specific Issues

En Windows usar `rmdir /s /q node_modules`; en macOS/Linux usar `rm -rf node_modules`.

### `Cannot read property 'X' of undefined` durante build

**Síntoma:** Error difuso durante `yarn run compile`.

**Solución:**
```bash
# 1. Verificar variables de entorno requeridas
cat .env.development    # deben estar definidas según config/env.js

# 2. Limpiar y recompilar
yarn run build-css
yarn run compile
```

---

### Bundle demasiado grande (memory error)

**Síntoma:**
```
FATAL ERROR: Ineffective mark-compacts near heap limit
```

**Solución:**
```bash
# Aumentar memoria para el proceso de build
NODE_OPTIONS="--max-old-space-size=4096" yarn run compile
```

---

### Webpack build genera archivos corruptos

**Síntoma:** Los chunks en `build/static/js/` están vacíos o incompletos.

**Solución:**
```bash
# Verificar que no haya compilaciones en paralelo
pm2 status   # asegurarse de que stvapp no esté sirviendo mientras compilas

# Limpiar y recompilar
rm -rf build/
yarn run build
```

### Componente se renderiza en bucle infinito

  }
}

    // ← SIEMPRE necesario en Smart TV
    clearInterval(this.interval);
    window.removeEventListener('keydown', this.handleKey);

  }
}

## 🔄 Flujos de Procesos Principales

### Flujo 1: Inicialización de la Aplicación

**Descripción:** Bootstrap completo desde `src/index.js` hasta renderizado de la SPA.

1. `src/index.js` detecta el dispositivo con `Device.getDevice()`
2. Si el dispositivo está en `FORCED_STALE_DEVICES` → redirige a dominio stale
3. Se verifica y configura `HKS` (token) desde URL params en localStorage
4. Se inicializa `AAF` (Advanced Advertising Framework)
5. Se monta `<Provider store>` con el Redux store
6. Se renderiza el componente raíz `App` desde `containers/App`

**Fuente:** `src/index.js:1-96`

### Flujo 2: Reproducción de Contenido (VOD / Live)

**Descripción:** Desde la selección de contenido hasta la reproducción en `AAFPlayer`.

1. Usuario selecciona contenido → dispatch de `playmedia.js` action
2. Redux actualiza `player` reducer con datos del contenido
3. Componente `Video/` o `playerDomain/` recibe nuevas props
4. `AAFPlayer.setup()` se inicializa con el wrapper HTML
5. `AAFPlayer` decide entre Bitmovin Player o Shaka Player según configuración
6. Eventos de player (load, play, pause, error) se emiten vía `AAFPlayerEvent`
7. Callbacks actualizan Redux store con estados de reproducción

**Fuente:** `src/AAFPlayer/AAFPlayer.js`, `src/actions/playmedia.js`, `src/reducers/player.js`

### Flujo 3: Autenticación de Usuario

**Descripción:** Login, gestión de tokens y sesión.

1. Usuario ingresa credenciales en `authentication/Login/`

3. `src/requests/Ajax.js` o `RequestManager.js` llama a API de auth

### Flujo 5: Control Parental (PIN)

**Descripción:** Verificación de PIN para contenido restringido.

1. Usuario intenta acceder a contenido para adultos/restringido
2. `AAF.Pin` activa flujo de verificación
3. Modal de PIN se muestra vía `pinCodes` reducer
4. Usuario ingresa PIN → validación en `src/actions/pinCodes.js`
5. Si válido → contenido desbloqueado; si inválido → mensaje de error
