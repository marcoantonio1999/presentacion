### Principios de Diseño

1. **Separación de responsabilidades** — cada módulo tiene una única responsabilidad: `:requestmanager` gestiona HTTP, `:imagemanager` gestiona imágenes. No hay dependencias cruzadas entre módulos SDK. **Evidencia:** `settings.gradle` — los módulos no se referencian entre sí.

2. **Abstracción vía interfaces** — los consumidores del SDK trabajan contra `RequestManagerInterface` e `ImageContract`, no contra implementaciones concretas. Esto permite cambiar el motor subyacente (Volley → Retrofit, Picasso → Glide/Coil) sin romper el API público. **Evidencia:** `RequestManagerInterface.java`, `ImageContract.java`.

3. **WeakReference para Context** — todo almacenamiento de `Context` en `RequestTask` se hace mediante `WeakReference<Context>` para prevenir memory leaks. **Evidencia:** `RequestTask.java` — campo `#WeakReference<Context> context`.

4. **⚠️ Singleton sin thread-safety** — ambos módulos implementan Singleton sin `volatile` ni double-checked locking, lo que puede generar instancias duplicadas en entornos multi-thread. **Evidencia:** `RequestManager.java:7`, `ImageManager.java:24`.

## 📦 Estructura de Módulos

### Módulos Core

> ⚠️ **No existe un módulo `:common` compartido.** Las clases utilitarias `L.java` y `CustomTimingLogger.java` están duplicadas en ambos módulos SDK, lo que constituye deuda técnica documentada.

**Evidencia:** `requestmanager/src/main/java/com/amco/requestmanager/L.java`, `imagemanager/src/main/java/amco/com/imagemanager/L.java` — archivos idénticos en funcionalidad.

### Módulos de Dominio

El dominio del SDK es la **abstracción de transporte**: encapsula cómo se hacen las peticiones HTTP y cómo se cargan las imágenes, de forma que el consumidor no interactúa directamente con Volley, Picasso ni OkHttp.

- **`:requestmanager`** — dominio: `RequestTask<F,E,T>` (Template Method) define el ciclo de vida completo de una petición. El consumidor solo implementa `processResponse()` y `useCache()`.
- **`:imagemanager`** — dominio: jerarquía de contratos (`ImageContract → ViewImageContract → AbstractImageContract → GenericImageContract + 12 contratos especializados`). El consumidor construye un contrato y llama a `displayImage()`.

**Evidencia:** `RequestTask.java:97`, `imagemanager/src/main/java/amco/com/imagemanager/contracts/` — 13 contratos verificados.

## 🔄 Patrones de Diseño

### Patrón Arquitectónico (MVVM, Clean, etc.)

**Abstract Factory** — `RequestManager` es una factory abstracta que instancia `VolleyRequestManager` por defecto. Permite intercambiar la implementación de red sin que el consumidor cambie su código. **Evidencia:** `RequestManager.java:8-14`.

**Template Method** — `RequestTask<F,E,T>` define el algoritmo completo de un request (validación de URL, caché lookup, llamada HTTP, parsing, callback). El consumidor solo implementa los métodos abstractos `processResponse()`, `useCache()` y opcionalmente `getMethod()`, `getHeaders()`, `getPostParams()`. **Evidencia:** `RequestTask.java:97`.

### Dependency Injection

> ⚠️ **El proyecto no usa ningún framework de DI** (Dagger, Hilt, Koin). Las dependencias se gestionan manualmente mediante:
> - Métodos factory estáticos: `RequestManager.getInstance()`, `ImageManager.getInstance()`
> - Constructores: `RequestTask(Context context)` — acepta `Context` via constructor
> - `WeakReference<Context>` — evita memory leaks al almacenar contexto

**Evidencia:** `RequestTask.java` — `WeakReference<Context>`, `RequestManager.java` — `getInstance()` sin parámetros.

### Repository Pattern

> ⚠️ **El patrón Repository no está implementado en el SDK.** `RequestTask` es el punto más cercano: encapsula el acceso a una fuente de datos remota, pero no abstrae múltiples fuentes (local + remoto). No hay capa Repository formal.

### Observer/State Management

El SDK usa un modelo de callbacks (Observer ligero) para comunicar el resultado de una operación al consumidor:

| Evento | Callback | Hilo de ejecución |
|--------|----------|-------------------|
| Request exitoso | `OnRequestListenerSuccess.onSuccess(result)` | Main thread |
| Request fallido | `OnRequestListenerFailed.onFailed(throwable)` | Main thread |
| Request cancelado | `onCancelled()` | — |
| Imagen cargada | `ImageContractListener.onSuccess(bitmap)` | Main thread |

**Evidencia:** `RequestTask.java:55-72`, `imagemanager/src/main/java/amco/com/imagemanager/interfaces/ImageContractListener.java`

## 🛡️ Consideraciones de Seguridad

### Autenticación y Autorización

> ⚠️ **El SDK no implementa ningún mecanismo de autenticación.** `RequestManagerInterface` expone `addRequest()` sin gestión de tokens ni sesiones. El consumidor debe gestionar autenticación externamente (sobrescribiendo `getHeaders()` en su subclase de `RequestTask`).

**Evidencia:** `RequestManagerInterface.java`, `RequestTask.java` — `getHeaders()` retorna `null` por defecto.

### Manejo de Secrets

> ⚠️ **No hay gestión de secrets en el SDK.** No se detectaron tokens, API keys ni credenciales hardcoded en el código fuente. Los consumidores son responsables de inyectar credenciales a través de los headers de `RequestTask`.

**Evidencia:** Revisión de todos los archivos `.java` — sin strings sensibles detectados.

### Validación de Datos

El único mecanismo de validación implementado en el SDK es la comprobación de URL vacía o nula en `RequestTask`:

```java
// RequestTask.java — lanzado si this.url es null o vacío
throw new ParamsException("Empty url request");
```

No hay validación de la respuesta del servidor ni sanitización de inputs. **Evidencia:** `ParamsException.java`, `RequestTask.java`.

### Protección de APIs

**Vulnerabilidad 1 — SSL Bypass en `:imagemanager`:**
`ImageManager.java:94-128` implementa un `TrustManager` que acepta **todos los certificados SSL** y un `HostnameVerifier` que acepta **cualquier hostname**. Ambos métodos son invocados en `init()` — el flujo de inicialización obligatorio.

```java
// ⛔ NO COPIAR — ImageManager.java:94-128
private SSLSocketFactory getAllTrustingSslSocketFactory() {
    // TrustManager.checkServerTrusted() VACÍO = acepta CUALQUIER certificado
}
private HostnameVerifier getAllTrustingHostnameVerifier() {
    return (hostname, session) -> true; // acepta CUALQUIER hostname
}
```

**Vulnerabilidad 2 — AsyncTask deprecated en `:requestmanager`:**
`RequestTask.java:237` extiende `AsyncTask`, eliminado en **API 33 (Android 13)**. El SDK no puede compilar para Android 13+ sin refactoring.

**Evidencia:** `imagemanager/src/main/java/amco/com/imagemanager/ImageManager.java:94-128`, `requestmanager/src/main/java/com/amco/requestmanager/RequestTask.java:237`

**Plan de remediación:** `.github/docs/03-diagnostico/SECURITY_REMEDIATION_PLAN.md`

## 📊 Performance y Escalabilidad

### Estrategias de Caching

**`:requestmanager`** — Volley gestiona una caché HTTP en memoria. La clave de caché es `method:url`. El comportamiento es controlable por el consumidor mediante `useCache()` e `isForceCache()` en cada `RequestTask`.

**`:imagemanager`** — Picasso implementa caché en dos niveles:
1. **Memoria** (`LruCache`) — tamaño configurable en `ImageManager.init()` (parámetro `memoryCacheSize`)
2. **Disco** — directorio y tamaño máximo configurables en `ImageManager.init()` (parámetros `cacheDir`, `diskCacheSize`)

**Evidencia:** `VolleyRequestManager.java`, `PicassoImageLoaderManager.java`

### Manejo de Concurrencia

| Mecanismo | Módulo | Estado |
|-----------|--------|--------|
| `Volley ThreadPool` (interno) | `:requestmanager` | ✅ Funcional |
| `AsyncTask` | `:requestmanager` | 🔴 Deprecated API 30, **eliminado API 33** |
| `OkHttp dispatcher` (interno a Picasso) | `:imagemanager` | ✅ Funcional |
| Singleton sin `volatile` | Ambos | ⚠️ Riesgo de instancias duplicadas en multi-thread |

**Evidencia:** `RequestTask.java:237` — `ProcessAsyncTask extends AsyncTask`, `RequestManager.java:7` — sin `volatile`

### Métricas de Performance

## 🔧 Evolución y Mantenimiento

El SDK está diseñado para ser extendido por el consumidor mediante herencia y composición:
- **Nuevos tipos de request** → extender `RequestTask<F,E,T>` e implementar `processResponse()`

## 🔄 Flujo de Datos

### Data Flow Diagram

**Flujo HTTP (`:requestmanager`):**

```
Consumidor
    │ crea subclase de RequestTask (define url, tag, processResponse)
    ▼
RequestManager.getInstance()
    │ retorna VolleyRequestManager (Abstract Factory)
    ▼
VolleyRequestManager.addRequest(requestTask)
    │ verifica caché (Volley DiskBasedCache)
    ├─ [CACHE HIT]──────────────────────────────────────────────────────┐
    │                                                                   │
    │ [CACHE MISS]                                                      │
    ▼                                                                   │
VolleyRequestTask (adapta RequestTask → Volley Request)                 │
    ▼                                                                   │
Volley RequestQueue → Red → HTTP Response                               │
    ▼                                                                   │
VolleyRequestTask.parseNetworkResponse()                                │
    ▼                                                                   │
RequestTask.processResponse() [background thread]                       │
    ▼                                                                   ▼
OnRequestListenerSuccess.onSuccess(resultado) [main thread] ◄──────────┘
```

**Flujo de carga de imagen (`:imagemanager`):**

```
Consumidor
    │ construye contrato: new GenericImageContract.Builder().setUrl().setImageView().build()
    ▼
ImageManager.getInstance()
    │ retorna PicassoImageLoaderManager (Abstract Factory)
    ▼
PicassoImageLoaderManager.displayImage(contract)
    │ extrae configuración del contrato
    ▼
Picasso.load(url).into(imageView)
    ├─ [LRU HIT] → ImageView.setImageBitmap() inmediato
    ├─ [DISK HIT] → decodifica → ImageView.setImageBitmap()
    └─ [CACHE MISS] → OkHttp3 → CDN → Picasso → ImageView.setImageBitmap()
```

**Evidencia:** `VolleyRequestManager.java`, `PicassoImageLoaderManager.java`, `RequestTask.java`

### State Management

> ⚠️ **No existe gestión de estado global en el SDK.** El SDK no mantiene estado de negocio entre requests. El único estado persistente son los Singletons (`RequestManager.requestManager`, `ImageManager.sInstance`) y la caché de Volley/Picasso.

**Evidencia:** `RequestManager.java:7` — `protected static RequestManager requestManager` sin `volatile`; `ImageManager.java:24` — `private static ImageManager sInstance` sin `volatile`.

### Event Handling

**Evidencia:** `RequestTask.java:55-72`, `imagemanager/src/main/java/amco/com/imagemanager/interfaces/`

El SDK comunica eventos mediante el patrón Callback (Observer simplificado):

| Evento | Interface | Hilo |
|--------|-----------|------|
| Request exitoso | `OnRequestListenerSuccess.onSuccess(E result)` | Main thread |
| Request fallido | `OnRequestListenerFailed.onFailed(Throwable error)` | Main thread |
| Request cancelado | `onCancelled()` | — |
| Imagen cargada | `ImageContractListener.onSuccess(Bitmap bitmap)` | Main thread |
| Imagen fallida | `NoViewImageContractListener.onFailed()` | — |

### Communication Patterns

**Evidencia:** `RequestManagerInterface.java`, `ImageContract.java`, contratos en `imagemanager/`

| Patrón | Componente | Descripción |
|--------|-----------|-------------|
| Abstract Factory | `RequestManager`, `ImageManager` | Instanciación de implementaciones sin exponer la clase concreta |

### Adding New Features

```java
// 1. Extender RequestTask con los tipos genéricos correctos
public class MiNuevoRequest extends RequestTask<String, MiModelo, Void> {

        super(context);

        return new Gson().fromJson(response, MiModelo.class);
    }

    @Override
    public boolean useCache() { return true; }

    @Override
    public int getMethod() { return Method.GET; }
}
```

**Principio:** Los consumidores del SDK trabajan contra interfaces (`RequestManagerInterface`, `ImageContract`), no contra clases concretas. Modificar las clases concretas sin cambiar las interfaces no rompe la compatibilidad del API público.

**Antes de modificar cualquier clase:**
1. Verificar si la clase es parte del API público (implementa una interface pública)
2. Si es API público: incrementar versión minor en `build.gradle` del módulo
3. Si es implementación interna: puede modificarse sin cambiar versión
4. Ejecutar `./gradlew :modulo:testDebugUnitTest` para verificar que los tests pasan
5. Ejecutar `./gradlew checkstyle` para verificar estilo

```java
    public MiNuevoRequest(Context context) {
```

**Solución:** Usar JDK 11 para ejecutar tests. JDK 17 puede usarse para compilación pero no para PowerMock.

**Evidencia:** `requestmanager/build.gradle` — `testImplementation 'org.powermock:powermock-module-junit4:1.6.4'`

**Causa:** `jcenter()` fue dado de baja en 2022.  
**Solución:** Agregar `mavenCentral()` en `build.gradle` raíz:

```groovy
allprojects {
    repositories {
        google()
        mavenCentral()  // Agregar esta línea
        jcenter()       // Mantener temporalmente como fallback
    }
}
```

**Evidencia:** `build.gradle` raíz — `jcenter()` actualmente en uso

#### Error: `Could not find com.squareup.picasso:picasso:2.71828`
**Causa:** Misma razón — jcenter obsoleto.  
**Solución:** Igual que el caso anterior — agregar `mavenCentral()`.

### Configuration Errors

#### Error: `Build Tools revision 28.0.3 is too old`
**Causa:** Build Tools 28.0.3 es muy obsoleto.  
**Solución:** Actualizar en `build.gradle` de cada módulo:

```groovy
android {

}
```

**Evidencia:** `requestmanager/build.gradle`, `imagemanager/build.gradle` — `buildToolsVersion '28.0.3'`

**Causa:** `android.support.v4.*` en `:requestmanager` conflictúa con AndroidX del proyecto consumidor.  
**Solución:** Agregar en `gradle.properties` del proyecto consumidor:

### Build Failures

**Causa más común:** Import no disponible o clase de `android.support.*` no encontrada.  
**Diagnóstico:** Revisar el stack trace completo en el output de Gradle.

#### Error: `Checkstyle violation`
**Causa:** Violación de reglas en `checkstyle.xml` (wildcard imports, nombres incorrectos, tabs).  
**Solución:**

## 🏃‍♂️ Runtime Issues

### Application Crashes

#### `ParamsException: Empty url request`
**Causa:** `this.url` es `null` o vacío en el constructor del `RequestTask`.  
**Solución:** Verificar que la URL se asigne antes de llamar a `addRequest()`:

```java
public MiRequest(Context context) {
    super(context);
    this.url = "https://api.example.com/endpoint"; // REQUERIDO
}
```

**Evidencia:** `ParamsException.java`, `VolleyRequestTask.java`

#### `NullPointerException` en `ImageManager`
**Causa más común:** `ImageManager.getInstance()` llamado antes de `init(context, ...)`.  
**Solución:** Inicializar en `Application.onCreate()`:

```java
@Override
public void onCreate() {
    super.onCreate();
    ImageManager.getInstance().init(this, ...);
}
```

### Performance Problems

**Síntoma:** Requests lentos o timeouts frecuentes con `TimeoutError`.  
**Causa posible 1:** Timeout default (45 segundos) insuficiente para el endpoint.  
**Solución:**

```java
@Override
public int getConnectTimeout() {
    return 60000; // 60 segundos
}
```

**Causa posible 2:** Muchos requests en cola sin cancelación.  
**Solución:** Usar `tag` para agrupar requests y cancelarlos al destruir la Activity:

```java
// En onDestroy()
RequestManager.getInstance().cancel(TAG);
```

**Evidencia:** `VolleyRequestManager.java`, `RequestManagerInterface.java`

### Memory Leaks

#### `OutOfMemoryError` en `imagemanager`
**Causa:** Cache de imágenes usando demasiada memoria (default: 1/3 de la RAM disponible).  
**Solución:** Ajustar `memoryCacheSize` al inicializar:

```java
ImageManager.getInstance().init(
    context,
    50 * 1024 * 1024,   // 50 MB memory cache
    cacheDir,
    100 * 1024 * 1024L, // 100 MB disk cache
    sslSocketFactory,
    hostnameVerifier
);
```

#### Leak de `Context` en `RequestTask`
**Correcto:** El proyecto ya usa `WeakReference<Context>` en `RequestTask`.  
**Verificar:** No pasar `Activity.this` directamente; usar `getApplicationContext()` cuando sea posible.

**Evidencia:** `RequestTask.java` — `WeakReference<Context>`

### Network Issues

#### Cache no funciona como se espera
**Síntoma:** Request siempre va a red aunque `useCache()` retorne `true`.  
**Diagnóstico:**

```java
String cacheKey = method + ":" + url;
Cache.Entry cacheData = queue.getCache().get(cacheKey);
L.d("Cache check: " + cacheKey + " → " + (cacheData != null ? "HIT" : "MISS"));
```

**Causas posibles:**
1. `isForceCache()` retorna `true` — revisar implementación del `RequestTask`
2. Headers del servidor desactivan el cache (no-cache, no-store)
3. El cache key incluye parámetros que cambian en cada request

**Evidencia:** `VolleyRequestManager.java:55-110`

#### Imágenes no cargan / errores SSL
**Causa conocida:** `ImageManager` usa SSL bypass inseguro (acepta todos los certificados).  
**⚠️ NO es un bug a resolver con un workaround** — es una vulnerabilidad de diseño documentada.  
**Ver:** `SECURITY_REVIEW.md` y `SECURITY_REMEDIATION_PLAN.md` para el plan de corrección.

**Evidencia:** `ImageManager.java:94-128` — `getAllTrustingSslSocketFactory()`

## 🧪 Testing Problems

### Test Failures

#### Tests fallan con `Method ... not mocked`
**Causa:** Tests unitarios de Android requieren `returnDefaultValues = true`.  
**Estado:** Ya configurado en los módulos del proyecto.  
**Verificar:** Confirmar en `build.gradle` del módulo:

```groovy
testOptions {
    unitTests {
        returnDefaultValues = true
    }
```

**Evidencia:** `requestmanager/build.gradle`, `imagemanager/build.gradle`

#### Tests de `VolleyRequestManager` fallan con `NullPointerException`
**Causa:** `RequestQueue` no inicializado en el test.  
**Solución:** Mockear `RequestQueue` con Mockito:

```java
@Mock RequestQueue mockQueue;
@Before
public void setUp() {
    MockitoAnnotations.initMocks(this);
    manager = new VolleyRequestManager(mockQueue);
}
```

**Evidencia:** `requestmanager/src/test/` — estructura de tests existentes

**Síntoma:** Tests que pasan a veces y fallan otras — especialmente tests que involucran threads.  
**Causa:** `AsyncTask` deprecated y comportamiento no determinista en tests unitarios.  
**Solución temporal:** Usar `@RunWith(RobolectricTestRunner.class)` para tests con Android looper, o mockear completamente los componentes asíncronos con Mockito.

**Evidencia:** `RequestTask.java` — uso de `AsyncTask` (deprecated en API 30)

### Test Environment Issues

**Solución a corto plazo:** Configurar JDK 11 en Android Studio para ejecutar tests.

`File → Project Structure → SDK Location → JDK Location → JDK 11`

**Cobertura estimada actual:** ~5% (11 tests / ~3,333 LOC main)  
**Solución:** Agregar en `requestmanager/build.gradle` e `imagemanager/build.gradle`:
