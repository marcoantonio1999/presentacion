### Componentes Principales

**Presentation Layer** — `com.amco.clarovideo.presentation`

Responsabilidades: renderizar la UI en Android TV (Leanback + View XML), manejar eventos D-pad/control remoto, observar `StateFlow`/`LiveData` de ViewModels, delegar toda lógica al ViewModel.

Componentes clave:
- `ClaroApp.kt` — Application class con `@HiltAndroidApp`
- `BaseActivity.kt` — Activity base para toda la app
- `BaseFragment.kt` — Fragment base con utilidades comunes
- `[Feature]Fragment.kt` — Implementaciones específicas por feature
- `[Feature]ViewModel.kt` — Estado y lógica de presentación

Nota: algunos Fragments heredan de `RowsSupportFragment` (Leanback) — confirmado en `SelectPlansDetailsFragment`, `PlayerEpisodeSectionsFragment`. `BrowseSupportFragment` **no se usa** en este proyecto.

**Domain Layer** — `com.amco.clarovideo.domain` + `com.amco.clarovideo.network.domain`

Responsabilidades: reglas de negocio, interfaces de repositorios, UseCases, modelos/entidades del dominio. La capa Domain **no conoce** Android SDK, Retrofit, Room ni ningún framework externo.

**Data Layer** — `com.amco.clarovideo.data` + `com.amco.clarovideo.network.data`

Responsabilidades: implementar interfaces de repositorio, gestionar fuentes de datos (red + BD local), manejar DTOs y conversión a entidades de dominio.

### Flujo de Datos

```
Usuario (D-pad / Remote)
    ↓
Fragment / Activity (UI Layer)
    ↓ observa StateFlow / LiveData
ViewModel (@HiltViewModel)
    ↓ invoca
UseCase (Domain)
    ↓ llama a interface
Repository Interface (Domain) ← contrato
    ↓ implementado por
RepositoryImpl (Data Layer)
    ├── Retrofit API (red)
    └── Room DAO (base de datos local)
         ├── LiveDatabase (LiveTV, v8)
         └── StartupAPIsDatabase (Config, v21)
```

## 🔄 Patrones de Diseño

### Patrón Arquitectónico (MVVM, Clean, etc.)

Clean Architecture de 3 capas con MVVM en la capa de presentación. Las dependencias fluyen exclusivamente hacia adentro: Presentation → Domain ← Data.

### Dependency Injection

Dagger Hilt 2.52 — Fuente: `build.gradle:23`

```kotlin
@HiltAndroidApp
class ClaroApp : Application()

@AndroidEntryPoint
class HomeFragment : Fragment() {
    private val viewModel: HomeViewModel by viewModels()
}

@HiltViewModel
class HomeViewModel @Inject constructor(
    private val getHomeRailsUseCase: GetHomeRailsUseCase
) : ViewModel()

class GetHomeRailsUseCase @Inject constructor(
    private val repository: HomeRepository
) {
    suspend operator fun invoke(): Flow<Result<List<HomeRail>>> =
        repository.getHomeRails()
}

@Module
@InstallIn(SingletonComponent::class)
object RepositoryModule {
    @Provides @Singleton
    fun provideHomeRepository(api: HomeApi, dao: HomeDao): HomeRepository =
        HomeRepositoryImpl(api, dao)
}
```

Nota: `@AndroidEntryPoint` se aplica selectivamente. Fragments sin inyección (ej. `TermsAndConditionsFragment`) **no lo usan** — Fuente: análisis de paquetes `settings/` y `register/`.

### Repository Pattern

```kotlin
// Interface en domain:
interface HomeRepository {
    fun getHomeRails(): Flow<Result<List<HomeRail>>>
}

// Implementación en data:
class HomeRepositoryImpl @Inject constructor(
    private val api: HomeApi,
    private val dao: HomeDao
) : HomeRepository { ... }

// Result pattern — Fuente: app/src/main/java/com/amco/clarovideo/data/Result.kt
sealed class Result<out T> {
    data class Success<T>(val data: T) : Result<T>()
    data class Error(val exception: Exception) : Result<Nothing>()
    object Loading : Result<Nothing>()
}
```

### Observer/State Management

| Mecanismo | Cuándo usar | Observaciones |
|-----------|-------------|---------------|
| `MutableStateFlow` | Estado de UI en ViewModels | **Preferido** en nuevos desarrollos |
| `StateFlow` | Exposición pública del estado | Immutable para observers |
| `LiveData` / `MutableLiveData` | Código legado | **Migrar** a StateFlow |
| `SharedFlow` | Eventos one-shot (navegación, toast) | Para efectos secundarios |
| `Flow` | Streams de datos en UseCase/Repo | Con `collect` en ViewModel |

## 🏗️ Arquitectura y Diseño

### Principios SOLID

- **Single Responsibility:** Cada UseCase hace exactamente una operación de negocio — Fuente: patrón `invoke()` en UseCases del proyecto
- **Open/Closed:** Repository interfaces como contratos extensibles sin modificar código existente
- **Dependency Inversion:** ViewModel depende de interfaces de UseCase, no de implementaciones

### Separación de Responsabilidades

```kotlin
// ✅ CORRECTO: ViewModel llama al UseCase
class HomeViewModel @Inject constructor(
    private val getHomeRailsUseCase: GetHomeRailsUseCase
) : ViewModel() {
    fun loadHome() {
        viewModelScope.launch { getHomeRailsUseCase().collect { ... } }
    }
}

// ❌ INCORRECTO: Fragment accede directo al Repository
class HomeFragment : Fragment() {
    @Inject lateinit var repository: HomeRepository // ❌ Violación de capas
}
```

### Manejo de Dependencias

```kotlin
// ✅ CORRECTO — @Inject constructor
class GetVodDetailUseCase @Inject constructor(
    private val repository: VodRepository
)

// ❌ INCORRECTO — instanciar manualmente
val useCase = GetVodDetailUseCase(repository)  // ❌ No usar new/constructor manual

// ✅ CORRECTO — KSP en módulos nuevos (NO kapt)
ksp "com.google.dagger:hilt-android-compiler:$dagger_hilt_version"
// ❌ INCORRECTO:
kapt "com.google.dagger:hilt-android-compiler:$dagger_hilt_version"  // ❌ kapt está deprecado

// ✅ CORRECTO — DataStore en lugar de SharedPreferences
class PreferencesDataStore @Inject constructor(
    private val dataStore: DataStore<Preferences>
)
// ❌ INCORRECTO — SharedPreferences
val prefs = context.getSharedPreferences("prefs", MODE_PRIVATE)  // ❌ Deprecated
```

### Patrones Recomendados

- **Coroutines + Flow** para nuevos desarrollos asíncronos — `viewModelScope.launch` y `lifecycleScope`
- **StateFlow** para estado de UI (no `MutableStateFlow` expuesto)
- **Result pattern** (`sealed class Result<T>`) para manejo de éxito/error/loading — Fuente: `data/Result.kt`
- **`GlobalScope` nunca** — usar siempre `viewModelScope` o `lifecycleScope`

## ⚠️ Anti-patrones

### Qué Evitar

| Anti-patrón | Problema | Alternativa |
|-------------|---------|-------------|
| Lógica en Fragment/Activity | Viola Clean Architecture | Mover al ViewModel/UseCase |
| `GlobalScope` | Memory leaks | Usar `viewModelScope` |
| `SharedPreferences` | Thread-unsafe, deprecated | DataStore |
| Hardcoded URLs/Keys | Seguridad crítica | BuildConfig + CI secrets |
| `kapt` en módulos nuevos | Lento, deprecated | `ksp` |
| Mezclar RxJava + Coroutines directo | Bugs sutiles | Usar adaptadores |
| Instanciar dependencias manualmente | Viola DI | `@Inject constructor` |
| `onTouchListener` en Android TV | No funciona en TV | D-pad handlers |
| `Log.d/e` en producción | Performance + seguridad | Timber con trees |
| `TELIPUtils` en código de producción | R8 renombra métodos privados | Solo usar en tests |

### Code Smells Comunes

- Exposición de `MutableStateFlow` o `MutableLiveData` como propiedad pública
- `withContext(Dispatchers.IO)` directamente en Fragments (debe estar en ViewModel/UseCase)
- Lógica condicional de negocio en Adapters o Presenters
- Tests comentados masivamente — evidenciado en `TopSectionViewHolderTest`, `VodDetailsCyclicFragmentTest`

### Ejemplos de Mal Código

- Mover `getCDN()` al ViewModel como función privada o al UseCase como lógica de dominio
- Mover `withContext(Dispatchers.IO)` al UseCase o Repository que ya está en el scope correcto
- Mover credenciales hardcodeadas (`authpn`, `authpt`, `session_key_ott_region`) de `QueryParamsConfig.kt` a `BuildConfig` fields configurados por flavor
