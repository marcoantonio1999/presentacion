## 🔴 P1 — SEC-04: Verificar New Relic Insert Key en `nr.js`

**Fuente:** `Clarovideo/js/nr.js` — función `sendErrorMetric()`
**OWASP:** A02 — Security Misconfiguration
**Severidad:** 🔴 Alto (condicional — depende del contenido real de `nr.js`)

### Descripción del Riesgo

La URL del collector de New Relic incluye el Account ID `675452` (identificador público, no sensible). Sin embargo, la autenticación real con New Relic Insights se realiza mediante un **Insert Key** enviado como header HTTP (`X-Insert-Key`). Si este key está hardcodeado en `nr.js`, cualquier persona con acceso al repositorio podría ingestar datos falsos en la cuenta de New Relic de Claro Video.

### Acción de Remediación

#### Paso 1: Auditar el contenido de `nr.js`

Verificar si el header `X-Insert-Key` está hardcodeado:

```javascript
// ⚠️ PATRÓN DE RIESGO — si existe algo como esto en nr.js:
xhr.setRequestHeader('X-Insert-Key', 'NRII-xxxxxxxxxxxxxxxxxxxx');

// ✅ PATRÓN SEGURO — el key debería abstraerse:
// Opción A: Externalizar a cnts.js con comentario de WARNING
var NR_INSERT_KEY = 'NRII-xxxxxxxxxxxxxxxxxxxx'; // WARNING: rotar periódicamente

// Opción B: Ofuscar clave para dificultar extracción del WGT
// (no es cifrado real, pero eleva la barra de extracción)
```

#### Paso 2: Si el Insert Key está hardcodeado

Dado que el proyecto es **Vanilla JS sin build process**, las opciones para proteger el key son limitadas. Plan recomendado:

| Opción | Descripción | Viabilidad | Seguridad |
|--------|-------------|------------|-----------|
| **A — Mantener en código (estado actual)** | El key es parte del WGT firmado. El WGT está cifrado y firmado por Samsung. Extracción requiere acceso físico al dispositivo + herramientas forenses. | ✅ Sin cambios | 🟡 Aceptable para TV App |
| **B — Rotar el Insert Key periódicamente** | Generar nuevo key en consola New Relic y actualizar `cnts.js` antes de cada release. Agregar al proceso de build. | ✅ Bajo esfuerzo | 🟡 Mejor que A |
| **C — Usar New Relic License Key (read-only)** | Si solo se necesita ingestar eventos (no leer), el License Key tiene permisos más restrictivos. | ✅ Bajo esfuerzo | 🟢 Recomendado |
| **D — Proxy de New Relic en el servidor AAF** | La app nativa envía eventos a un endpoint del servidor AAF, que los reenvía a New Relic con el key guardado en el servidor. | ⚠️ Alto esfuerzo — requiere cambios en backend | 🟢 Máxima seguridad |

**Recomendación:** Implementar Opción B + C de inmediato. Evaluar Opción D para la próxima versión mayor.

#### Paso 3: Agregar al proceso de release

Agregar al `package_prod.sh` una verificación de que el Insert Key no es el valor de desarrollo:

```bash
# Verificación de seguridad: el NR key de producción debe ser diferente al de dev
# Agregar a package_prod.sh antes del tizen package
echo "✅ Verificación de seguridad: confirmar que NR_INSERT_KEY en cnts.js es el de PRODUCCIÓN"
grep -n "NR_INSERT_KEY\|X-Insert-Key" Clarovideo/js/nr.js Clarovideo/js/cnts.js
```

**Tiempo estimado:** 2-4 horas (auditoría) + 1-2 horas (rotación de key si necesario)

## 🟡 P2 — SEC-01: `allow.mixedcontent` en `config.xml`

**Fuente:** `Clarovideo/config.xml`
**OWASP:** A02 — Security Misconfiguration
**Severidad:** 🟡 Medio

### Descripción del Riesgo

`allow.mixedcontent: true` permite que el WebView del TV cargue recursos HTTP incluso cuando el protocolo principal es HTTPS. En producción, `cnts.js` fuerza HTTPS (`app_http_protocol = "https:"`), pero si el servidor AAF devuelve recursos HTTP embebidos, el WGT no los bloqueará.

### Acción de Remediación

#### Opción A: Mantener para builds HTTP (estado actual justificado)

El flag existe específicamente para el build `package_http.sh` (usado con Charles proxy). La arquitectura de 3 builds (`prod`, `dev`, `http`) fue diseñada para esto.

**Riesgo residual:** El mismo WGT con `allow.mixedcontent: true` se usa en producción. Si se controla el servidor AAF para no devolver mixed content, el riesgo es aceptable.

#### Opción B: Separar el flag por ambiente de build (RECOMENDADO)

Modificar los scripts de build para generar `config.xml` con el flag correcto por ambiente:

```bash
# En package_prod.sh y package_dev.sh — deshabilitar mixedcontent
sed -i 's/allow.mixedcontent.*true/allow.mixedcontent type="boolean">false/' Clarovideo/config.xml

# Compilar...

# Restaurar para builds http
# (en package_http.sh mantener true)
```

**Precaución:** Verificar que el app hosteado (React) no depende de mixed content para none de sus recursos antes de deshabilitar.

**Tiempo estimado:** 4-8 horas (verificación con equipo del app hosteado + modificación de scripts)

## 🟡 P4 — SEC-06: Verificación de integridad del bundle solo por disponibilidad HTTP

**Fuente:** `Clarovideo/js/appManager.js`
**OWASP:** A08 — Software and Data Integrity Failures
**Severidad:** 🟡 Medio

### Descripción del Riesgo

El preflight check actual verifica:
1. Que el servidor AAF devuelve HTTP 200 para `index.html`
2. Que el bundle JS (`main.[hash].js`) existe y devuelve HTTP 200

### Acción de Remediación

Agregar headers `Content-Security-Policy` con hashes SHA-256 de los bundles críticos. Requiere cambios en el servidor AAF y en el proceso de deployment del app React.

```javascript
// En appManager.js — verificación de hash criptográfico del contenido (requiere Crypto API)
// ⚠️ Verificar disponibilidad de SubtleCrypto en Tizen 3.0 antes de implementar
```

#### Opción B: Validación de tamaño mínimo de respuesta (corto plazo, sin costo)

Agregar una verificación mínima del tamaño del bundle en `appManager.js`:

```javascript
// En el callback de éxito del XHR del hash.js
// Si el bundle JS tiene < 1KB de contenido, algo está mal
if (xhr.responseText && xhr.responseText.length < 1000) {
```

## 🟢 P5 — SEC-05: `innerHTML` en `nativeModal.js` con contenido estático

**Fuente:** `Clarovideo/js/nativeModal.js`
**OWASP:** A05 — Injection
**Severidad:** 🟢 Bajo

### Descripción del Riesgo

`nativeModal.js` usa `innerHTML` para construir el HTML del modal. El contenido viene de `cnts.js` (constantes compiladas en el WGT), por lo que no hay riesgo de XSS en el estado actual.

El riesgo sería si en el futuro el contenido del modal incluyera texto proveniente de respuestas del servidor (por ejemplo, mensajes de error del servidor AAF).

### Acción de Remediación

#### Plan preventivo (sin cambio urgente)

Agregar un comentario de advertencia en `nativeModal.js` para futuros desarrolladores:

```javascript
// SEGURIDAD: Se usa innerHTML para construir el modal.
// El contenido proviene EXCLUSIVAMENTE de cnts.js (constantes compiladas en el WGT).
// NO agregar aquí texto proveniente de respuestas del servidor o input del usuario
// sin aplicar sanitización primero (usar textContent en lugar de innerHTML para strings externos).
```

Si en el futuro se agrega texto dinámico al modal, reemplazar la asignación `innerHTML` con `textContent` para el texto variable:

```javascript
// ✅ Seguro para texto externo
element.textContent = mensajeDelServidor; // sanitiza automáticamente

// ⚠️ Solo para HTML estático del bundle
element.innerHTML = htmlEstaticoDeCnts; // solo si viene de cnts.js
```
