### Componentes Principales

**Presentation Layer** ÔÇö `com.amco.clarovideo.presentation`

Responsabilidades: renderizar la UI en Android TV (Leanback + View XML), manejar eventos D-pad/control remoto, observar `StateFlow`/`LiveData` de ViewModels, delegar toda lógica al ViewModel.

Componentes clave:
- `ClaroApp.kt` ÔÇö Application class con `@HiltAndroidApp`
- `BaseActivity.kt` ÔÇö Activity base para toda la app
- `BaseFragment.kt` ÔÇö Fragment base con utilidades comunes
- `[Feature]Fragment.kt` ÔÇö Implementaciones específicas por feature
- `[Feature]ViewModel.kt` ÔÇö Estado y lógica de presentación

Nota: algunos Fragments heredan de `RowsSupportFragment` (Leanback) ÔÇö confirmado en `SelectPlansDetailsFragment`, `PlayerEpisodeSectionsFragment`. `BrowseSupportFragment` **no se usa** en este proyecto.

**Domain Layer** ÔÇö `com.amco.clarovideo.domain` + `com.amco.clarovideo.network.domain`

Responsabilidades: reglas de negocio, interfaces de repositorios, UseCases, modelos/entidades del dominio. La capa Domain **no conoce** Android SDK, Retrofit, Room ni ningún framework externo.

**Data Layer** ÔÇö `com.amco.clarovideo.data` + `com.amco.clarovideo.network.data`

Responsabilidades: implementar interfaces de repositorio, gestionar fuentes de datos (red + BD local), manejar DTOs y conversión a entidades de dominio.

## 🔄 Patrones de Diseño

### Patrón Arquitectónico (MVVM, Clean, etc.)

Clean Architecture de 3 capas con MVVM en la capa de presentación. Las dependencias fluyen exclusivamente hacia adentro: Presentation Ôćĺ Domain ÔćÉ Data.

### Dependency Injection

Dagger Hilt 2.52 ÔÇö Fuente: `build.gradle:23`

```kotlin
@HiltAndroidApp
class ClaroApp : Application()

@AndroidEntryPoint
class HomeFragment : Fragment() {
    private val viewModel: HomeViewModel by viewModels()
}

@HiltViewModel
class HomeViewModel @Inject constructor(
    private val getHomeRailsUseCase: GetHomeRailsUseCase
) : ViewModel()

class GetHomeRailsUseCase @Inject constructor(
    private val repository: HomeRepository
) {
    suspend operator fun invoke(): Flow<Result<List<HomeRail>>> =
        repository.getHomeRails()
}

@Module
@InstallIn(SingletonComponent::class)
object RepositoryModule {
    @Provides @Singleton
    fun provideHomeRepository(api: HomeApi, dao: HomeDao): HomeRepository =
        HomeRepositoryImpl(api, dao)
}
```

Nota: `@AndroidEntryPoint` se aplica selectivamente. Fragments sin inyección (ej. `TermsAndConditionsFragment`) **no lo usan** ÔÇö Fuente: análisis de paquetes `settings/` y `register/`.

### Repository Pattern

```kotlin
// Interface en domain:
interface HomeRepository {
    fun getHomeRails(): Flow<Result<List<HomeRail>>>
}

// Implementación en data:
class HomeRepositoryImpl @Inject constructor(
    private val api: HomeApi,
    private val dao: HomeDao
) : HomeRepository { ... }

// Result pattern ÔÇö Fuente: app/src/main/java/com/amco/clarovideo/data/Result.kt
sealed class Result<out T> {
    data class Success<T>(val data: T) : Result<T>()
    data class Error(val exception: Exception) : Result<Nothing>()
    object Loading : Result<Nothing>()
}
```

### Observer/State Management

| Mecanismo | Cuándo usar | Observaciones |
|-----------|-------------|---------------|
| `MutableStateFlow` | Estado de UI en ViewModels | **Preferido** en nuevos desarrollos |
| `StateFlow` | Exposición pública del estado | Immutable para observers |
| `LiveData` / `MutableLiveData` | Código legado | **Migrar** a StateFlow |
| `SharedFlow` | Eventos one-shot (navegación, toast) | Para efectos secundarios |
| `Flow` | Streams de datos en UseCase/Repo | Con `collect` en ViewModel |

## 🔧 Evolución y Mantenimiento

### Principios de Extensibilidad

- Agregar nuevas features siguiendo el patrón `presentation/[feature]/` + `domain/[feature]/` + `data/[feature]/`
- Toda nueva dependencia debe evaluarse frente al stack existente (evitar triplicar librerías como ocurre con imágenes y JSON)
- Prefiera Coroutines sobre RxJava para nuevas implementaciones

### Versionado de APIs

- API version actual: `v5.94` en `QueryParamsConfig.kt` y `PaymentsUtils.kt`
- Build variants controlan los application IDs por entorno: `dev Ôćĺ com.amx.atv` / `prod Ôćĺ com.amco.clarovideo_atv`

### Estrategias de Migración

- Migraciones Room mediante `object : Migration(from, to)` ÔÇö ver patrón en sección Bases de Datos
- Al modificar entidades Room: incrementar `version` en `@Database` y agregar migración correspondiente en `AMXNetworkSdk/schemas/`

### Qué Evitar

| Anti-patrón | Problema | Alternativa |

| Lógica en Fragment/Activity | Viola Clean Architecture | Mover al ViewModel/UseCase |

## 🚀 Introducción
