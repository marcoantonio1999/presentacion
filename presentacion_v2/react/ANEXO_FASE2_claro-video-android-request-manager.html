## 1. ¿Qué es AmcoLibs y por qué importa?

AmcoLibs es la biblioteca Android que usa la aplicación Claro Video para cargar imágenes y gestionar las comunicaciones de red hacia los servidores. Está integrada directamente en la aplicación que los usuarios descargan del Play Store.

Cualquier problema de seguridad en esta biblioteca afecta **directamente a todos los usuarios activos de Claro Video** que tienen la aplicación instalada.

---

## 2. Resumen Ejecutivo — Hallazgos Críticos

### 🔴 Riesgo Crítico — Exposición de Datos de Usuarios

Se encontró código en la biblioteca que **desactiva deliberadamente la verificación de seguridad de las conexiones HTTPS**.

**En términos simples:** Esto permite que cualquier persona en la misma red Wi-Fi que un usuario (por ejemplo, en un café, aeropuerto, o red corporativa) intercepte y lea el tráfico de red de la aplicación. Esto incluye potencialmente credenciales, tokens de sesión, y datos de usuario.

**Evidencia técnica:** `ImageManager.java`, líneas 91–136 — método `getAllTrustingSslSocketFactory()` con verificación de certificados vacía.

**Clasificación:** OWASP M3 — Insecure Communication | CVSS estimado: 8.1/10 (ALTO)

## 4. ¿Qué se encontró?

### Problema 1: Seguridad Crítica (Impacto: Usuarios finales)
- La biblioteca desactiva la verificación de certificados SSL/TLS
- Código aparentemente de desarrollo/debugging que llegó a producción
- **Resultado:** Usuarios vulnerables a ataques de interceptación de red (Man-in-the-Middle)

### Problema 2: Sin Infraestructura de Pruebas (Impacto: Calidad del producto)
- No existe ningún test automatizado ejecutable
- No hay pipeline de integración continua
- **Resultado:** Cada cambio al SDK puede romper la aplicación sin que haya detección automática

### Problema 3: Dependencias Obsoletas (Impacto: App Store + Seguridad)
- Se usa `android.support.v4` que Google dejó sin mantenimiento en 2019
- Se usa Android Gradle Plugin 4.2.2 (versión actual: 8.x)
- **Resultado:** Posible rechazo del Play Store; vulnerabilidades conocidas sin parche disponible

### Problema 4: Código Duplicado (Impacto: Costo de mantenimiento)
- Dos módulos del SDK contienen clases idénticas (`L.java`, `CustomTimingLogger.java`)
- **Resultado:** Correcciones deben hacerse dos veces; mayor probabilidad de inconsistencias

## 📐 Características ISO/IEC 5055

La norma ISO/IEC 5055 define 4 características de calidad del software medibles a partir del análisis estático:

1. **Reliability** — Confiabilidad: Capacidad de realizar funciones bajo condiciones establecidas
2. **Security** — Seguridad: Protección contra acceso no autorizado y explotación
3. **Performance Efficiency** — Eficiencia de Rendimiento: Recursos utilizados para ejecutar funciones
4. **Maintainability** — Mantenibilidad: Facilidad de modificación y evolución del sistema

> **Nota metodológica:** ISO/IEC 5055 original requiere análisis automatizado con herramientas certificadas (e.g., CAST AIP). Este reporte aplica la metodología desde análisis estático manual del código fuente. Las puntuaciones son estimaciones fundadas.

## 🔴 1. Reliability (Confiabilidad)

**Score:** 45/100 — 🟠 RIESGO ALTO

### Hallazgos

| ID | Hallazgo | Archivo | Línea(s) | Impacto ISO 5055 |
|----|----------|---------|----------|-----------------|
| REL-01 | `getInstance()` de `ImageManager` sin `synchronized` — race condition posible en inicialización concurrente | `ImageManager.java` | ~134–138 | ❌ Reliability flaw — concurrent access |
| REL-02 | `static mRequestQueue` en `VolleyRequestManager` sin sincronización — state corruption en apps multi-thread | `VolleyRequestManager.java` | 15–17 | ❌ Reliability flaw — shared mutable state |

### Análisis

La confiabilidad del SDK está comprometida por dos patrones de thread-safety deficientes. En aplicaciones Android de producción, los `ImageManager` y `VolleyRequestManager` son frecuentemente inicializados desde `Application.onCreate()` o fragmentos de UI concurrentes. La ausencia de sincronización en los singletons puede resultar en `NullPointerException` intermitentes difíciles de reproducir.

**Puntaje negativo principal:** REL-01 y REL-02 son fallas estructurales graves. REL-03 suma riesgo de comportamiento indefinido en el target predominante (Android 11+ > 60% del mercado).

## 🔴 2. Security (Seguridad)

**Score:** 10/100 — 🔴 CRÍTICO

### Hallazgos

| ID | Hallazgo | Archivo | Línea(s) | OWASP | ISO 5055 |

### Análisis

El hallazgo SEC-01 + SEC-02 representa una falla de seguridad **crítica activa en producción**: toda la infraestructura de carga de imágenes del SDK opera con SSL completamente deshabilitado. El comentario en el código (`// Create a trust manager that does not validate certificate chains`) confirma que se implementó intencionalmente para desarrollo, pero no se revirtió antes de producción.

Específicamente, el vector de ataque es:
1. Un atacante en la misma red (WiFi público, café, aeropuerto) puede ejecutar ARP spoofing
2. Intercept el tráfico HTTPS del dispositivo
3. El SDK aceptará el certificado falso del atacante sin error
4. El atacante puede leer todos los headers HTTP de las peticiones de imagen (posiblemente conteniendo tokens de autenticación) e inyectar respuestas maliciosas

**Este único hallazgo coloca al SDK en la categoría CRÍTICO para ISO/IEC 5055 Security.**

## 🔴 Riesgos Críticos

### RIESGO-01 — Bypass SSL/TLS Total
| Campo | Dato |
|-------|------|
| **ID** | RIESGO-01 |
| **Componente** | `imagemanager` |
| **Archivo** | `imagemanager/src/main/java/amco/com/imagemanager/ImageManager.java:91–136` |
| **Probabilidad** | **Alta** — El código bypass está activo en producción; cualquier build usa esta configuración |
| **Impacto** | **Crítico** — Permite MITM completo, robo de credenciales/tokens transmitidos via imagen URLs |
| **Nivel de Riesgo** | 🔴 **CRÍTICO** |
| **OWASP** | A02:2021 — Cryptographic Failures |
| **ISO/IEC 5055** | Reliability (4.1), Security (4.4) |

**Descripción técnica:**
- `getAllTrustingSslSocketFactory()` (líneas 91–127): `X509TrustManager` con métodos `checkServerTrusted()` y `checkClientTrusted()` **vacíos** → acepta cualquier certificado SSL, incluyendo expirados, auto-firmados y de autoridades comprometidas.
- `getAllTrustingHostnameVerifier()` (líneas 129–136): retorna `true` para **cualquier hostname** → un atacante puede presentar certificado válido de otro dominio.
- El comentario literal en el código: `// Create a trust manager that does not validate certificate chains` — confirma que la intención era desarrollo/debug pero el código llegó a producción.
- `ImageManager.init()` (línea 142) invoca ambos métodos → **toda carga de imágenes** en la aplicación queda expuesta.

**Vector de ataque:**
```
Usuario → ARP Poison / Rogue WiFi AP → MITM Proxy 
    → Intercepta requests de imagen de Claro Video
    → Obtiene URLs/tokens/cookies de sesión en headers HTTP
    → Inyecta contenido malicioso en respuestas de imagen
```

**Mitigación requerida:**
- Eliminar `getAllTrustingSslSocketFactory()` y `getAllTrustingHostnameVerifier()`
- Implementar `CertificatePinner` de OkHttp o `android:networkSecurityConfig` con `<pin-set>`
- Ver `SECURITY_REMEDIATION_PLAN.md` para plan detallado

### RIESGO-05 — Thread-Safety Ausente en Singletons
| Campo | Dato |
|-------|------|
| **ID** | RIESGO-05 |
| **Componente** | `:requestmanager` + `:imagemanager` |
| **Archivos** | `VolleyRequestManager.java:15–17` (campos `static` no sincronizados), `ImageManager.java` (getInstance sin sincronización) |
| **Probabilidad** | **Media** — Requiere inicialización concurrente desde múltiples threads simultáneos |
| **Impacto** | **Alto** — `NullPointerException` o estado corrupto en producción, difícil de reproducir |
| **Nivel de Riesgo** | 🟠 **ALTO** |
| **ISO/IEC 5055** | Reliability (4.1) |

**Descripción:**
- `VolleyRequestManager.mRequestQueue` es `static` y se inicializa condicionalmente sin `synchronized`.
- `ImageManager.getInstance()` no implementa `synchronized`, `volatile` ni initialization-on-demand holder.
- En apps con múltiples threads (common en Android), dos threads pueden ejecutar el branch de inicialización simultáneamente.

**Mitigación:**
- Implementar Initialization-on-Demand Holder para `ImageManager`
- Sincronizar `VolleyRequestManager` con `synchronized`/`volatile` o refactorizar a inyección de dependencias

X509TrustManager trustAllCerts = new X509TrustManager() {

### Análisis de la Vulnerabilidad

Los métodos `checkClientTrusted()` y `checkServerTrusted()` están **completamente vacíos**. Esto significa:

1. **checkServerTrusted vacío:** El SDK no verifica si el certificado presentado por el servidor es:
   - Emitido por una Autoridad Certificadora (CA) confiable
   - Válido (no expirado)
   - Perteneciente al dominio correcto
   - No revocado

2. **Resultado:** Un atacante puede presentar cualquier certificado — incluso uno auto-firmado en 5 minutos — y el SDK lo aceptará sin error.

3. **Amplitud del impacto:** `ImageManager.init()` (línea 142) invoca `getAllTrustingSslSocketFactory()` e instancia el `OkHttpClient` con este TrustManager → **absolutamente todas las peticiones de imagen** del SDK operan con SSL deshabilitado.

### Vector de Ataque

```
Atacante
  │
  ├── Conecta a misma red WiFi que la víctima (café, aeropuerto, hotel)
  │
  ├── Ejecuta ARP poisoning: herramienta arp-scan + arpspoof
  │      → Redirige tráfico de víctima a través de su máquina
  │
  ├── Intercepta conexión HTTPS hacia CDN de Claro Video
  │      → Víctima: "Cargando imagen de https://cdn.clarovideo.com/..."
  │      → SDK: presenta certificado falso del atacante
  │      → ImageManager.checkServerTrusted(): vacío → sin error → acepta
  │
  ├── Lee en claro:
  │      → Headers HTTP: Authorization, Cookie, X-Auth-Token
  │      → Metadatos de contenido solicitado (revela historial de visualización)
  │
  └── Inyecta:
         → Imágenes reemplazadas por contenido malicioso
         → HTML/Script en respuestas con Content-Type incorrecto
```

- **Privacidad:** Un atacante puede conocer qué contenido está viendo el usuario (URLs de imágenes revelan series/películas)
- **Autenticación:** Si tokens de sesión se transmiten en headers de imagen-requests, pueden ser robados

### Análisis

## 🔥 Hotspots de Deuda Técnica

### 🔴 CRÍTICO — Prioridad Inmediata

#### 1. SSL Bypass Total — `ImageManager.java`
- **Archivo:** `imagemanager/src/main/java/amco/com/imagemanager/ImageManager.java`
- **Líneas:** 91–127 (`getAllTrustingSslSocketFactory`) + 129–136 (`getAllTrustingHostnameVerifier`)
- **Descripción:** El SDK acepta **cualquier certificado SSL**, incluyendo certificados inválidos, caducados o auto-firmados. El `X509TrustManager` tiene `checkServerTrusted()` y `checkClientTrusted()` **completamente vacíos**. El `HostnameVerifier` retorna `true` para **cualquier hostname**.
- **Impacto de seguridad:** OWASP A02:2021 — Cryptographic Failures. Permite ataques Man-in-the-Middle (MITM) contra cualquier request de imagen.
- **Deuda estimada:** 4 horas — Implementar `CertificatePinner` de OkHttp o configurar `NetworkSecurityConfig`
- **Componente afectado:** Security (30% del score)

#### 2. Cobertura de Tests Desconocida
- **Archivos:** `requestmanager/src/test/`, `imagemanager/src/test/`, `imagemanager/src/androidTest/`
- **Descripción:** Directorios de test existen pero no hay reportes Jacoco, no hay CI/CD configurado y no hay métricas de cobertura disponibles en el repositorio.
- **Impacto:** Todo refactoring del SDK introduce riesgo de regresión sin red de seguridad medible.
- **Deuda estimada:** 2–3 días — Configurar Jacoco + ejecutar tests existentes + agregar tests faltantes
- **Componente afectado:** TestCoverage (15% del score) — actualmente 0

### 🟡 ALTO — Planificar en Sprint Actual

#### 3. `AsyncTask` Deprecado — `RequestTask.java`
- **Archivo:** `requestmanager/src/main/java/com/amco/requestmanager/RequestTask.java:5`
- **Descripción:** `import android.os.AsyncTask` — deprecado desde Android API 30 (Android 11). Puede producir comportamiento inesperado en dispositivos modernos.
- **Deuda estimada:** 8–12 horas — Migrar a `Executor` + `Handler` o Kotlin Coroutines (requiere migración de lenguaje)
- **Componente afectado:** CodeQuality

#### 4. `android.support.v4` Sin Migrar a AndroidX
- **Archivo:** `requestmanager/build.gradle:10` — `com.android.support:support-v4:28.0.0`
- **Descripción:** La librería `android.support.v4` está en End-of-Life desde 2018. Google no publica parches de seguridad para esta versión. AndroidX es el reemplazo oficial.
- **Deuda estimada:** 4–6 horas — Migrar con `Refactor > Migrate to AndroidX` en Android Studio
- **Componente afectado:** CodeQuality + Security

#### 5. `targetSdkVersion 26` y `compileSdkVersion 28` Desactualizados
- **Archivo:** `requestmanager/build.gradle:20-26`, `imagemanager/build.gradle`
- **Descripción:** `targetSdkVersion 26` (Android 8.0) impide publicación en Google Play Store (mínimo requerido: API 33 desde agosto 2023). `compileSdkVersion 28` no recibe actualizaciones de seguridad de Android.
- **Deuda estimada:** 1–2 días — Actualizar a `compileSdk 35`, `targetSdk 35`
- **Componente afectado:** Architecture + CodeQuality

### 🟢 MEDIO — Planificar en Próximo Sprint

#### 6. Código Duplicado Entre Módulos
- **Archivos duplicados:**
  - `requestmanager/.../CustomTimingLogger.java` ↔ `imagemanager/.../CustomTimingLogger.java`
  - `requestmanager/.../L.java` ↔ `imagemanager/.../L.java`
- **Descripción:** Clases utilitarias idénticas en ambos módulos. Cambiar una no garantiza cambiar la otra.
- **Deuda estimada:** 2 horas — Extraer a módulo compartido `:common` o `:utils`
- **Componente afectado:** Architecture

#### 7. Singletons Sin Thread Safety
- **Archivos:**
  - `ImageManager.java:134-138` — `getInstance()` sin `synchronized`
  - `VolleyRequestManager.java` — campos `static` no sincronizados
- **Descripción:** En aplicaciones multi-threaded, múltiples threads pueden crear instancias simultáneas (race condition) o leer estado inconsistente.
- **Deuda estimada:** 1–2 horas — Implementar `volatile + double-checked locking` o `Holder pattern`
- **Componente afectado:** Architecture + CodeQuality

#### 8. TODO Residuales y Typos en Código de Producción
- **Archivo:** `requestmanager/src/main/java/com/amco/requestmanager/RequestTask.java:28,33,38`
- **Descripción:** 3× `// TODO Create public access methods to change to private accesor` (typo: "accesor" debería ser "accessor"). Indica campos `protected` que deberían ser `private` con getters/setters.
- **Deuda estimada:** 1 hora — Encapsular campos + corregir typos
- **Componente afectado:** CodeQuality + Documentation

#### 9. Dependencias de Testing Muy Antiguas
- **Archivo:** `requestmanager/build.gradle:13-16`
- **Descripción:**
  - Mockito `1.10.19` (2014) → actual 5.x
  - PowerMock `1.6.4` (2016) → no recomendado (usar mockito-inline)
  - JUnit `4.12` → JUnit 5 es el estándar actual
- **Deuda estimada:** 4 horas — Actualizar dependencias de test
- **Componente afectado:** CodeQuality + TestCoverage

#### 10. Documentación Javadoc Parcial
- **Archivos:** todos los 39 archivos Java en `src/main/`
- **Descripción:** Solo `VolleyRequestManager.java` tiene Javadoc parcial. Las 26 clases de `imagemanager/` y 11 clases de `requestmanager/` (excluyendo VolleyRequestManager) carecen de documentación de API pública.

- **Componente afectado:** Documentation

        PILM-->>App: imagen mostrada en ImageView
