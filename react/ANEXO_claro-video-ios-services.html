## 📋 Resumen Arquitectónico

CVServices es una librería Swift Package Manager con **5 módulos independientes** que proveen funcionalidad compartida (networking, modelos, vistas, extensiones, localización) para las apps iOS y tvOS de Claro Video. El paquete tiene **cero dependencias externas** — 100% Apple SDK nativo.

### Stack Tecnológico

| Categoría | Tecnología | Versión / Detalle |
|-----------|-----------|-------------------|
| Lenguaje | Swift | 5.7 (`swift-tools-version:5.7`) |
| Gestor de paquetes | SPM (Swift Package Manager) | Nativo con Xcode |
| Plataformas | iOS, tvOS, macOS | iOS 16+, tvOS 16+, macOS 13+ |
| Networking | URLSession | Nativo (via `NetworkDispatcher`) |
| Reactive | Combine | Uso mínimo (2 archivos: `PassthroughSubject` para token expiry) |
| UI declarativo | SwiftUI | 32 archivos (vistas compartidas iOS/tvOS) |
| UI imperativo | UIKit | 33 archivos (extensiones de componentes) |
| Testing | XCTest | 14 archivos de test |
| CI/CD | Azure DevOps | `ci/pipeline-analysis.yml` |
| Análisis estático | SonarCloud | Org: `clarovideo`, key: `cv-ios-services` |
| Repositorio | GitHub | Source control |

### Patrones Arquitectónicos Principales

1. **Service Layer Pattern (3 niveles):** Managers (facade) → Services (lógica) → NetworkDispatcher (HTTP)
2. **Protocol-Oriented Design:** Contratos formales (`CVServiceProtocol`, `RequestProtocol`, `DispatcherProtocol`)
3. **Service Locator:** `QueryBuilder.shared.getService(for:)` centraliza la creación de servicios
4. **Facade Pattern:** `*ServicesManager` exponen API pública estática simple
5. **Dual Parsing:** Coexistencia de `Codable` (60 archivos) y `parseObject(dictionary:)` manual (28 archivos)

### Principios de Diseño

- **Modularidad:** 5 módulos SPM con responsabilidades claras y dependencias explícitas
- **Zero dependencias externas:** Todo el código depende exclusivamente de Apple SDK (Foundation, URLSession, Combine, SwiftUI, UIKit)
- **Multi-plataforma:** Compilación condicional `#if os(iOS)` / `#if os(tvOS)` para adaptar constantes y comportamiento por plataforma
- **Encapsulamiento de servicios:** Las apps consumidoras interactúan solo con `*ServicesManager` (facade), nunca con servicios internos o `NetworkDispatcher` directamente
- **Configuración dinámica:** Datos de usuario, sesión y dispositivo se inyectan vía closures en `QueryBuilder.shared`, permitiendo configuración lazy y dinámica

### Flujo de Datos

```
App → ServicesManager.action(completion:)
  → QueryBuilder.shared.getService(for: .requestType) as DomainServices
    → DomainServices.action(completion:)
      → NetworkDispatcher(environment:).fetch(request:, baseParams:)
        → URLSession.shared.dataTask(with: urlRequest)
          → API REST Claro Video
        ← JSON Response (Data)
      ← Model (via Codable/JSONDecoder o parseObject/manual)
    ← Result<Model, NetworkError>
  ← completion(.success(model) / .failure(error))
```

**Detalle del flujo:**
1. La app consumidora llama a un método estático en `*ServicesManager`
2. El manager obtiene la instancia del servicio configurado vía `QueryBuilder.shared.getService(for:)`
3. El servicio construye el request usando `RequestProtocol` y lo pasa a `NetworkDispatcher`
4. `NetworkDispatcher` construye el `URLRequest`, agrega parámetros base del `QueryBuilder`, y ejecuta la llamada HTTP
5. La respuesta JSON se parsea al modelo correspondiente (Codable o manual)
6. El resultado (`Result<Model, NetworkError>`) se devuelve vía callback al caller original

## 🔄 Patrones de Diseño

### Patron Arquitectonico (MVVM, Clean, etc.)

El proyecto no usa MVVM, Clean Architecture ni VIPER formalmente. Su patrón principal es **Service Layer** de 3 niveles con diseño orientado a protocolos:

| Nivel | Carpeta | Responsabilidad | Ejemplo |
|-------|---------|----------------|---------|
| **Facade** | `CVManagers/` | API pública estática para apps consumidoras | `InitialServicesManager.getStartHeaderInfo(completion:)` |
| **Service** | `ClaroVideoServices/Service/` | Lógica de negocio + invocación de networking + parsing de respuesta | `InitialServices.getStartHeaderInfo(completion:)` |
| **Network** | `ClaroVideoNetworking/` | Ejecución HTTP con URLSession, manejo de errores, token expiry | `NetworkDispatcher.fetch(request:, baseParams:, completion:)` |

**Protocolos fundamentales:**

| Protocolo | Archivo | Propósito |
|-----------|---------|-----------|
| `CVServiceProtocol` | `ClaroVideoServices/Protocol/CVServiceProtocol.swift` | Contrato base que todo servicio debe cumplir (propiedades: `environment`, `baseParameters`) |
| `RequestProtocol` | `ClaroVideoNetworking/Protocol/RequestProtocol.swift` | Define cómo se construye un HTTP request (path, method, headers, body) |
| `DispatcherProtocol` | `ClaroVideoNetworking/Protocol/DispatcherProtocol.swift` | Abstracción de la capa de networking (permite sustituir URLSession) |

### Dependency Injection

No hay framework DI formal (Swinject, Factory, etc.). La inyección se realiza mediante:

1. **`Environment` struct:** Pasado al `NetworkDispatcher` con baseURL y configuración de red
2. **`ServiceParameters` dictionary:** Parámetros por request inyectados desde `QueryBuilder`
3. **Closures en `QBLocalData` y `QBUserData`:** Datos dinámicos via closures (ej: `userData.userToken?()`, `localData.hks?()`)
4. **Service Locator:** `QueryBuilder.shared.getService(for:)` actúa como factory de servicios configurados

```swift
// Ejemplo de inyección vía closures en QueryBuilder
QueryBuilder.shared.localData.hks = { return SessionManager.shared.currentHKS }
QueryBuilder.shared.userData.userToken = { return AuthManager.shared.token }

// Ejemplo de Service Locator
let service = QueryBuilder.shared.getService(for: .startHeaderInfo) as InitialServices
// service ya tiene environment y baseParameters inyectados
```

### Repository Pattern

⚠️ **Dato no disponible** — No se detectó una implementación formal del Repository Pattern en el repositorio. Los servicios (`*Services`) acceden directamente al `NetworkDispatcher` sin una capa intermedia de abstracción de datos. Los modelos en `CVModel` son structs/classes de datos puros sin lógica de persistencia local. No hay capa de almacenamiento local (CoreData, Realm, UserDefaults wrappers) en la librería.

### Observer/State Management

El manejo de estado reactivo es mínimo:

- **Combine (`PassthroughSubject`):** Usado en `NetworkDispatcher.isTokenExpired` para notificar cuando un token ha expirado. Las apps consumidoras se suscriben a este publisher:

```swift
// En NetworkDispatcher
public static let isTokenExpired = PassthroughSubject<Bool, Never>()

// En la app consumidora
NetworkDispatcher.isTokenExpired
    .sink { isExpired in
        if isExpired { /* renovar token o cerrar sesión */ }
    }
    .store(in: &cancellables)
```

- **Callbacks (`@escaping` closures):** Patrón predominante para comunicar resultados asíncronos:

```swift
public static func getStartHeaderInfo(completion: @escaping (Result<StartHeaderInfo, NetworkError>) -> Void)
```

- **No se usa:** `@Published`, `ObservableObject`, `@StateObject`, ni state management libraries (Redux, TCA, etc.) dentro de la librería. Las vistas en `CVCommonViews` reciben datos como parámetros de inicialización, no via bindings reactivos.

## 🛡️ Consideraciones de Seguridad

### Autenticación y Autorización

- **Tokens dinámicos:** User token, session key y CM access token se inyectan vía closures en `QBUserData`, evaluados al momento de cada request:
  ```swift
  QueryBuilder.shared.userData.userToken = { return currentToken }
  QueryBuilder.shared.userData.sessionKey = { return currentSessionKey }
  QueryBuilder.shared.userData.cmAccessToken = { return currentCMToken }
  ```
- **HKS (Hash Key Session):** Gestionado vía closure en `QBLocalData.hks` — se incluye como parámetro base en las requests
- **Token expirado:** `NetworkDispatcher.isTokenExpired` (Combine `PassthroughSubject`) emite notificación reactiva cuando el servidor reporta token inválido

### Manejo de Secrets

- No se detectaron secrets, API keys ni contraseñas hardcodeados en el código fuente
- Todos los valores sensibles (tokens, HKS, session keys) se reciben de la app consumidora vía closures en runtime
- El archivo `.copilotrc` y `.github/copilot-instructions.md` no contienen credenciales

### Validación de Datos

- **Parsing Codable:** Los modelos `Decodable` validan estructura JSON automáticamente — fallos de decodificación generan errores controlados
- **Parsing manual:** Los 28 archivos con `parseObject(dictionary:)` usan casting (`as?` / `as!`) — los 14 force unwraps (`as!`) son un riesgo de crash si la estructura JSON no coincide
- **`NetworkError`:** Enum de errores que incluye: `.badRequest`, `.noData`, `.unauthorized`, `.serverError`, entre otros

### Protección de APIs

- **HTTPS:** `HttpProtocol` enum incluye `.https` como protocolo disponible
- **Parámetros base:** `QueryBuilder` inyecta automáticamente parámetros de autenticación (HKS, token, región) en cada request
- **No se detectó:** Certificate pinning, request signing, ni rate limiting dentro de la librería

## 📊 Performance y Escalabilidad

### Estrategias de Caching

⚠️ **Dato no disponible** — No se implementa caching explícito dentro de CVServices. Se depende del caching HTTP nativo de `URLSession` (`URLCache`). Se recomienda evaluar caching a nivel de servicio para responses frecuentes (EPG, catálogo).

### Optimizaciones de Consultas

- **Parámetros dinámicos:** `QueryBuilder` construye parámetros base una sola vez por tipo de request, evitando reconstrucción innecesaria
- **URLSession shared:** Reutilización de la sesión HTTP compartida para todas las requests (connection pooling nativo de iOS)
- **Cancel support:** `NetworkDispatcher` soporta `task?.cancel()` para cancelar requests en vuelo

### Manejo de Concurrencia

- **Background dispatching:** Requests se ejecutan en `DispatchQueue.global(qos: .background)` para no bloquear el hilo principal
- **Callbacks en completion:** Los resultados se devuelven vía `@escaping` closures — el caller es responsable de dispatch al main thread para UI updates
- **No usa async/await:** A pesar de que Swift 5.7 lo soporta, todo el networking usa callbacks. Migración recomendada en [ACTION_PLAN.md](../04-accion/ACTION_PLAN.md)
- **No usa Structured Concurrency:** No se detectaron `Task`, `TaskGroup`, `Actor` ni `@Sendable` en el código

## 🔧 Evolución y Mantenimiento

### Principios de Extensibilidad

**Agregar un nuevo servicio** sigue un patrón repetible (5 archivos):

1. **`CVRequestType`** — Agregar caso al enum (ej: `.newDomainAction`)
2. **`NewDomainRequest`** — Crear struct que conforme `RequestProtocol` (path, method, headers)
3. **`NewDomainServices`** — Crear clase que conforme `CVServiceProtocol` (lógica + parsing)
4. **`NewDomainServicesManager`** — Crear clase facade con métodos estáticos públicos
5. **`QueryBuilder`** — Registrar el nuevo servicio en `getService(for:)` y `getDefaultParams(for:)`

**Agregar un nuevo modelo:**
1. Crear struct/class `Codable` en `CVModel/Models/NewDomain/`
2. Opcionalmente crear extensión `+Parceable` si se necesita compatibilidad con parsing legacy

## 🏗️ Arquitectura y Diseño

### Principios SOLID

| Principio | Aplicación en CVServices |
|-----------|-------------------------|
| **S — Single Responsibility** | Cada módulo SPM tiene una responsabilidad clara (networking, modelos, extensiones, localización, vistas). Cada `ServicesManager` maneja un solo dominio |
| **O — Open/Closed** | Nuevos servicios se agregan creando nuevas clases que conforman `CVServiceProtocol` sin modificar las existentes (excepto registrar en `QueryBuilder`) |
| **L — Liskov Substitution** | Todos los servicios son intercambiables vía `CVServiceProtocol`. `DispatcherProtocol` permite sustituir `NetworkDispatcher` |
| **I — Interface Segregation** | Protocolos separados: `CVServiceProtocol` (servicios), `RequestProtocol` (requests), `DispatcherProtocol` (red) |
| **D — Dependency Inversion** | Los servicios dependen de abstracciones (`DispatcherProtocol`, `RequestProtocol`), no de implementaciones concretas |

**Áreas de mejora SOLID:** `QueryBuilder.shared` (893 LOC) viola SRP al combinar Service Locator + configuración + parámetros base en un solo singleton.

### Separación de Responsabilidades

```
CVManagers/        → Solo facade (delegar a Services, NUNCA lógica de negocio aquí)
Services/          → Lógica de negocio + orquestación de parsing
Request/           → Solo definición de endpoint HTTP (path, method, headers)
NetworkDispatcher  → Solo ejecución HTTP y manejo de errores de red
QueryBuilder       → Solo configuración y factory de servicios
CVModel            → Solo definición de datos y parsing
```

**Regla clave:** Los `ServicesManager` deben ser **facades puras** — un método estático que llama al servicio correspondiente. No deben contener lógica de negocio, transformación de datos, ni manejo de errores custom.

### Manejo de Dependencias

- **SPM exclusivamente:** No usar CocoaPods ni Carthage
- **Cero dependencias externas:** Toda funcionalidad se implementa con Apple SDK nativo (Foundation, URLSession, Combine, SwiftUI, UIKit)
- **Dependencias internas unidireccionales:** `CVExtension` ← `CVLocalizable` ← `CVModel` ← `CVCommonViews` ← `CVServices` (sin ciclos)
- **Inyección vía closures:** Datos dinámicos (tokens, HKS, región) se inyectan via closures en `QueryBuilder.shared`, no como parámetros directos

```swift
// ✅ Inyección dinámica vía closure

// ❌ EVITAR: hardcodear valores
```

## ⚠️ Anti-patrones

### Qué Evitar

| Anti-Patrón | Por qué es Problema | Qué Hacer |
|-------------|---------------------|-----------|
| `as!` force unwrap | Crash en runtime si tipo no coincide | Usar `as?` con `guard let` / `if let` |
| `try!` force try | Crash si la operación falla | Usar `try?` o `do-catch` |
| Callbacks anidados | Pirámide de doom, difícil de leer/mantener | Extraer a funciones, o migrar a async/await |
| Código en `OldModels/` | Legacy, inconsistente con patrones actuales | No usar como referencia; crear nuevo en `Models/` |
| `OldServices/` | Servicios desactualizados | Implementar en `Service/` con `CVServiceProtocol` |
| Strings hardcoded | Difícil de localizar | Usar `CVLocalizable` |
| Lógica en Managers | Managers deben ser solo facade | Mover lógica a Services |
| `public` innecesario | Expone API interna | Usar `internal` por defecto |
| `print()` en producción | Contamina logs, posible leak de datos | Usar `os.Logger` o eliminar |

- Formato `Type+Funcionalidad.swift`: `Date+Formats.swift`, `Array+Utils.swift`, `Date+BussinessRules.swift`

- Archivo modelo: `Model.swift`

```swift
// 1. Imports
import Foundation

// MARK: - Initializer
// MARK: - Public Methods
// MARK: - Private Methods
```

**Organización por feature (Domain-Driven):**

## 🎯 Responsabilidades por Módulo

### CVServices: Responsabilidades

**Archivos:** 80 | **Dependencias:** CVExtension, CVModel, CVCommonViews

CVServices es el módulo principal y único producto exportado. Agrupa tres responsabilidades:

1. **CVManagers/ (Facade Layer):** 19 clases finales estáticas que exponen la API pública del SDK.
   - Patrón: `public static func action(completion: @escaping (Result<Model, NetworkError>) -> Void)`
   - Ejemplo: `InitialServicesManager.getStartHeaderInfo(completion:)`
   - Los managers NO contienen lógica — solo delegan al servicio correspondiente.

2. **ClaroVideoServices/ (Service + Request Layer):** 22 servicios + 20 requests + protocolos.
   - `CVServiceProtocol`: contrato base con métodos para cada acción del servicio.
   - `RequestProtocol`: define path, method, headers, parameters de cada request HTTP.
   - `OldServices/`: servicios legacy pendientes de migración (candidato a eliminación).

   - `QBLocalData`: HKS (Hash Key Session), región, idioma, sesión.
   - `QBUserData`: token, sessionKey, cmAccessToken (via closures para datos dinámicos).
   - `QBEndpoints`: URLs base de servicios.
   - `CVRequestType`: enum con ~80+ tipos de request.
   - `getService(for:)`: Service Locator que resuelve servicios por tipo de request.

```
│  CVModel/ (Parsing)                                              │
│                                                                  │
│  Codable:  JSONDecoder().decode(Model.self, from: data)          │
│  Manual:   Model.parseObject(dictionary: jsonDict)               │
│                                                                  │
│  Retorna: Result<Model, NetworkError>                            │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ 5. Resultado final
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  App Consumidora recibe Result<Model, NetworkError> en closure   │
│  completion(.success(model)) / completion(.failure(error))       │
└──────────────────────────────────────────────────────────────────┘
```

### State Management

El proyecto utiliza un enfoque **minimalista** para estado:

| Componente | Mecanismo de estado | Scope |
|------------|---------------------|-------|
| `QueryBuilder.shared` | Singleton mutable | Global (toda la app) |
| `QBDeviceData` | Struct configurable | Sesión de la app |
| `QBLocalData` | Closures dinámicos (`hks`, `region`) | Request-time |
| `QBUserData` | Closures dinámicos (`userToken`, `sessionKey`) | Request-time |
| `CVCommonViews` | `@State`, `@Binding` (SwiftUI local) | Vista individual |
| `PurchaseDetail` | ViewModel con `@Published` | Vista + sub-vistas |

> **Nota:** No hay framework de state management global (no Redux, no TCA). El estado se distribuye entre el singleton QueryBuilder y los closures inyectados por la app consumidora.

### Event Handling

**Combine (reactivo):**
- `NetworkDispatcher.isTokenExpired`: `PassthroughSubject<Bool, Never>` — emite `true` cuando el servidor rechaza un token.
- La app consumidora se suscribe a este subject para disparar flujo de re-autenticación.

**Callbacks (imperativo):**
- Patrón dominante: `completion: @escaping (Result<Model, NetworkError>) -> Void`
- Usado en todos los managers y servicios sin excepción.

**NotificationCenter:**
- ⚠️ Dato no disponible — No se identificó uso explícito de `NotificationCenter` en el análisis del código fuente de la librería. Las notificaciones se delegan a la app consumidora.

### Communication Patterns

| Patrón | Dónde se usa | Descripción |
|--------|-------------|-------------|
| **Callback/Closure** | Managers → App | `Result<Model, NetworkError>` en completion handlers |
| **Delegation (Protocol)** | Services → NetworkDispatcher | `CVServiceProtocol`, `RequestProtocol`, `DispatcherProtocol` |
| **Service Locator** | QueryBuilder → Services | `getService(for: CVRequestType)` resuelve servicios por tipo |
| **Singleton** | QueryBuilder.shared | Configuración global compartida entre servicios |
| **Reactive (Combine)** | NetworkDispatcher → App | `PassthroughSubject` para eventos de token expirado |
| **Environment** | NetworkDispatcher | Struct `Environment` inyectado con protocolo HTTP y base URL |

### Adding New Features

**Agregar un nuevo servicio (endpoint)** — 5 pasos:

1. **Definir Request Type** en `CVRequestType.swift`:
   ```swift
   // CVServices/QueryBuilder/CVRequestType.swift
   public enum CVRequestType {
       case nuevoServicio  // ← Agregar case
   }
   ```

2. **Crear Request** en `ClaroVideoServices/Request/`:
   ```swift
   // NuevoServicioRequest.swift
   struct NuevoServicioRequest: RequestProtocol {
       var path: String { "/services/nuevoservicio" }
       var method: HttpMethod { .get }
       var parameters: ServiceParameters { [:] }
   }
   ```

3. **Crear Modelo** en `CVModel/Models/NuevoDominio/`:
   ```swift
   // NuevoModelo.swift
   public struct NuevoModelo: Codable {
       public let id: String
       public let nombre: String
   }
   ```

4. **Crear Service** en `ClaroVideoServices/Service/`:
   ```swift
   // NuevoServicioService.swift
   final class NuevoServicioService: CVServiceProtocol {
       func obtenerDatos(completion: @escaping (Result<NuevoModelo, NetworkError>) -> Void) {
           let request = NuevoServicioRequest()
           dispatcher.execute(request: request) { result in
               // Parsear y retornar resultado
           }
       }
   }
   ```

5. **Crear Manager** en `CVManagers/`:
   ```swift
   // NuevoServicioManager.swift
   public final class NuevoServicioManager {
       private static func getService(requestType: CVRequestType) -> NuevoServicioService {
           return QueryBuilder.shared.getService(for: requestType) as NuevoServicioService
       }
       
       public static func obtenerDatos(completion: @escaping (Result<NuevoModelo, NetworkError>) -> Void) {
           getService(requestType: .nuevoServicio).obtenerDatos(completion: completion)
       }
   }
   ```

**Agregar una nueva vista compartida:**

1. Crear directorio en `CVCommonViews/NuevaVista/`
2. Crear `NuevaVista.swift` como `View` SwiftUI
3. Usar compilación condicional `#if os(iOS)` / `#if os(tvOS)` para valores de plataforma
4. Si necesita ViewModel, crear subdirectorio `ViewModel/` siguiendo el patrón de `PurchaseDetail`

### Modifying Existing Code

**Reglas para modificar código existente:**

1. **No modificar `OldServices/` ni `OldModels/`** — están marcados para eliminación. Crear reemplazos nuevos.
2. **Mantener compatibilidad de API pública** — los managers son la interfaz pública. Cambios internos no deben romper la firma de los managers.
3. **Respetar el patrón existente:**
   - Managers: siempre `public final class` con métodos `public static`
   - Services: siempre conforman `CVServiceProtocol`
   - Requests: siempre conforman `RequestProtocol`
   - Modelos: preferir `Codable` sobre parsing manual
4. **Compilación condicional:** Si el cambio afecta UI, verificar ambas plataformas:
   ```swift
   #if os(iOS)
   // Comportamiento iOS
   #endif
   #if os(tvOS)
   // Comportamiento tvOS — valores de UI suelen ser más grandes
   #endif
   ```
5. **Tests:** Agregar test unitario en `Tests/CVServicesTests/` para todo modelo nuevo o modificado.

**Deuda técnica identificada con prioridad de refactoring:**

### Migration Strategies

**Migración Codable (prioridad media):**
1. Identificar modelos con `parseObject(dictionary:)` en `CVModel/`
2. Agregar conformancia `Codable` al struct existente
3. Definir `CodingKeys` si las keys del JSON difieren de las propiedades
4. Crear tests con el mismo JSON fixture para validar equivalencia
5. Actualizar el servicio para usar `JSONDecoder` en lugar de `parseObject`
6. Deprecar `parseObject` con `@available(*, deprecated)`
7. Eliminar `parseObject` en la siguiente versión mayor

**Migración async/await (prioridad baja):**
1. Agregar variante `async` manteniendo el closure original:

   ```swift
       try await withCheckedThrowingContinuation { continuation in
           obtenerDatos { result in
               continuation.resume(with: result)
           }
       }
   }
   
   // Existente — mantener para retrocompatibilidad
   public static func obtenerDatos(completion: @escaping (Result<NuevoModelo, NetworkError>) -> Void) { ... }
   ```
2. Las apps consumidoras migran a su ritmo
3. Deprecar variante closure cuando todas las apps hayan migrado

**Migración de QueryBuilder (prioridad baja):**
1. Extraer `QBDeviceData`, `QBLocalData`, `QBUserData`, `QBEndpoints` a archivos separados
2. Convertir `QueryBuilder` en una fachada que compone los sub-componentes
3. Evaluar inyección de dependencias formal para reemplazar el singleton

```swift

    case action

    var method: HTTPMethod { .get }
    var headers: HTTPHeaders { [:] }
    var timeoutInterval: TimeInterval { 30 }
```

```swift
final class NewService: CVServiceProtocol {
    var environment: Environment
    var baseParameters: ServiceParameters?
    
    init(environment: Environment, baseParameters: ServiceParameters?) {
        self.environment = environment
        self.baseParameters = baseParameters
    }
    
    func performAction(completion: @escaping (Result<NewModel, NetworkError>) -> Void) {
        NetworkDispatcher(environment: environment)
            .fetchCommonResponse(request: NewServiceRequest.action, baseParams: baseParameters, completion: completion)
    }
}
```

```swift
public final class NewServicesManager {

        getService(requestType: .newServiceAction).performAction(completion: completion)
    }
}
```

## 🚀 Introducción

CVServices es una librería Swift distribuida como Swift Package Manager que provee servicios compartidos de networking, modelos de datos, vistas SwiftUI, extensiones comunes y localización para las aplicaciones iOS y tvOS de Claro Video. El paquete contiene 5 módulos independientes (CVServices, CVModel, CVExtension, CVLocalizable, CVCommonViews) con **cero dependencias externas** — 100% Apple SDK nativo. Soporta iOS 16+, tvOS 16+ y macOS 13+.

### Application Crashes

#### Crash en `parseObject(dictionary:)` — Force Unwrap

```swift
// ⚠️ Código problemático (14 instancias identificadas en CVModel)
let response = jsonResponse["response"] as! [String: AnyObject]
```

**Causa:** La API REST retornó una estructura JSON diferente a la esperada (campo faltante, tipo diferente, null).

**Solución inmediata:** Usar optional binding:
```swift
// ✅ Correcto
guard let response = jsonResponse["response"] as? [String: AnyObject] else {
    return .failure(.parsingError("Missing 'response' key"))
}
```

**Ubicaciones conocidas de force unwraps:** 14 `as!` en `CVModel/` — ver [DIAGNOSTIC_REPORT.md](../03-diagnostico/DIAGNOSTIC_REPORT.md) para lista completa.

#### Crash por `try!` (force try)

```swift

let data = try! JSONEncoder().encode(model)
```

**Solución:** Reemplazar con `do-catch` o `try?`:
```swift
do {
```

### Performance Problems

#### Requests bloqueantes en main thread

**Síntoma:** UI se congela durante llamadas de red.

**Causa técnica:** Aunque `NetworkDispatcher` despacha en `DispatchQueue.global(qos: .background)`, el completion handler puede ejecutarse en cualquier queue.

**Solución en la app consumidora:**
```swift
InitialServicesManager.getStartHeaderInfo { result in
    DispatchQueue.main.async {
        // Actualizar UI aquí
        switch result {
        case .success(let model): updateUI(model)
        case .failure(let error): showError(error)
        }
    }
}
```

#### QueryBuilder.shared — contención por singleton

**Síntoma:** Latencia inesperada cuando múltiples servicios ejecutan simultáneamente.

**Causa:** `QueryBuilder.shared` (893 LOC) es accedido concurrentemente sin sincronización explícita.

**Mitigación:** Evitar modificar propiedades de `QueryBuilder.shared` desde múltiples threads simultáneamente. Configurar una sola vez al inicio de la app.

### Memory Leaks

#### Retain cycles en completion handlers

**Síntoma:** Memoria no se libera después de cerrar un flujo que usa servicios.

**Patrón problemático:**
```swift
// ⚠️ Potencial retain cycle
class MyViewController {
    func loadData() {
        InitialServicesManager.getStartHeaderInfo { result in
            self.updateUI(result)  // strong reference a self
        }
    }
}
```

**Solución:**
```swift
// ✅ Captura débil
InitialServicesManager.getStartHeaderInfo { [weak self] result in
    self?.updateUI(result)
}
```

#### Combine subscriptions sin cancelación

**Patrón problemático para token expirado:**
```swift
// ⚠️ Subscription sin almacenar → se cancela inmediatamente
NetworkDispatcher.isTokenExpired.sink { _ in }

// ⚠️ O almacenada sin cancelar → leak
```

**Solución:**
```swift
var cancellables = Set<AnyCancellable>()

NetworkDispatcher.isTokenExpired
    .sink { [weak self] isExpired in
        if isExpired { self?.handleTokenExpired() }
    }
    .store(in: &cancellables)
// cancellables se limpia cuando el objeto se deinitializa
```

### Network Issues

#### "Token expired" sin handler — requests fallan silenciosamente

**Causa:** `NetworkDispatcher.isTokenExpired` emite evento via `PassthroughSubject<Bool, Never>` pero nadie está suscrito.

**Solución:** En la app consumidora, suscribirse al subject:
```swift
NetworkDispatcher.isTokenExpired
    .sink { isExpired in
        if isExpired {
            // Renovar token o redirigir a login
            AuthManager.refreshToken()
        }
    }
    .store(in: &cancellables)
```

#### Timeout en requests y sin respuesta

**Causa:** `URLSession.shared` usa timeout por defecto (~60s). Si la API de Claro Video no responde, el request se bloquea.

**Diagnóstico:**
```swift
// Verificar endpoints configurados
print(QueryBuilder.shared.endpoints)  // Solo para debugging

// Verificar conectividad al API
curl -v "https://api.clarovideo.com/services/health"
```

#### Errores HTTP no manejados

Los códigos HTTP se procesan en `NetworkDispatcher`. Errores comunes:

| Código | Significado | Acción |
|--------|------------|--------|
| 401 | Token expirado | `isTokenExpired` emite `true` — renovar token |
| 403 | Sin permisos | Verificar HKS y región |
| 404 | Endpoint no existe | Verificar `CVRequestType` y URL version |
| 500 | Error del servidor | Reintentar con backoff o reportar |
| 503 | Servicio no disponible | Reintentar más tarde |

## 🔄 Patrones Identificados

### Patrones Arquitectónicos

| Patrón | Implementación | Evidencia | Estado |
|--------|---------------|-----------|--------|
| **Layered Architecture** | 4 capas: Facade → Service → Request → Network | Estructura de carpetas `CVManagers/` → `Service/` → `Request/` → `Networking/` | 🟢 Bien implementado |
| **Service Locator** | `QueryBuilder.shared.getService(for:)` | `QueryBuilder.swift` (893 LOC) | 🟡 Funcional pero acoplante |
| **Facade Pattern** | 19 `*ServicesManager` como API pública | `CVManagers/*.swift` (19 archivos) | 🟢 Bien implementado |
| **Module Separation** | 5 targets SPM independientes | `Package.swift` | 🟢 Bien implementado |

### Patrones de Diseño

| Patrón | Implementación | Evidencia |
|--------|---------------|-----------|
| **Singleton** | `QueryBuilder.shared` como configuración global | `QueryBuilder.swift` línea ~15 |
| **Protocol-Oriented** | `CVServiceProtocol`, `RequestProtocol`, `DispatcherProtocol` | `ClaroVideoNetworking/`, `ClaroVideoServices/Service/` |
| **Factory Method** | `getService(for:)` crea servicios tipados | `QueryBuilder.swift` método `getService(for:)` |
| **Strategy** | Diferentes `*Request` para cada endpoint | 20 archivos en `ClaroVideoServices/Request/` |
| **Observer** | `PassthroughSubject<Bool, Never>` para token expirado | `NetworkDispatcher.swift` propiedad `isTokenExpired` |
| **Template Method** | Patrón común en managers: `get*Services()` → `service.action()` → `completion()` | 19 managers con estructura idéntica |

### Anti-patrones Detectados

| Anti-patrón | Ubicación | Impacto | Recomendación |
|-------------|-----------|---------|---------------|
| **God Object** | `QueryBuilder.shared` (893 LOC) | 🔴 Alto — concentra configuración, creación de servicios, parámetros default, lógica de platform | Descomponer en `ServiceFactory`, `ParameterBuilder`, `DeviceConfigurator` |
| **Service Locator** (como anti-patrón) | `QueryBuilder.shared.getService(for:)` | 🟠 Medio — oculta dependencias, dificulta testing | Migrar a Dependency Injection con protocolo `ServiceProviding` |
| **Parsing dual** | 60 `Codable` + 28 `parseObject` manual | 🟡 Medio — inconsistencia, duplicación de lógica | Migrar todos los modelos a `Codable`/`Decodable` |
| **Force Unwrap** | 14 ocurrencias de `as!` en parsers | 🔴 Alto — crash en runtime si el tipo no coincide | Reemplazar por `as?` con manejo de error |
| **Callback Hell** | Completion handlers anidados sin `async/await` | 🟡 Medio — código difícil de leer y mantener | Agregar wrappers `async/await` |
| **Dead Code** | `OldModels/`, `OldServices/` | 🟡 Medio — deuda técnica, confusión para nuevos devs | Eliminar tras verificar que no hay consumidores |
| **Naming Inconsistency** | `ResponceProtocol` (Response), `Parceable` (Parseable) | 🟢 Bajo — confusión menor | Renombrar con aliases de compatibilidad |
